Pocket Earth 口播稿：电台机制版
去除拉美作家例子，加入海明威与波拉尼奥电台示例。
我做了一个本地空间知识库，叫 Pocket Earth，口袋地球。
它第一眼看起来像一个地图应用。可我真正想做的，是把自己的书、电影、音乐、照片、行程，还有某一刻写下的心情，都钉回地球上。
我把地理坐标当成个人知识库的索引轴。
平时整理信息，我们常用链接、标签、文件夹，或者一个数据库。Pocket Earth 换了一种组织方式：每一条记录都尽量回答一个问题，它属于地球上的哪里。
一本书，可以落到故事发生地。
一部电影，可以落到取景地、国别，或者某个电影之都。
一首歌，可以落到歌手的出身城市，也可以落到歌词里写到的城市。
一张照片，本身就带着经纬度。
一段旅行走完，也会变成地球上一串只属于我的足迹。
最后，这些原本形状完全不同的东西，会被统一钉回同一颗地球。
我选择地理坐标，是因为人对地点的记忆往往很强。很多时候，我们可能忘了某件事具体是哪天发生的，也忘了文件叫什么名字，但会记得它发生在哪里。地点是一种很自然的记忆入口。
先说数据层。
现在项目里已经接入了几类真实数据。
电影是豆瓣导出的 2124 条观影记录。书是豆瓣导出的 1055 本阅读记录。音乐侧有 96 座城市、621 首歌，还有 619 位艺人的流派分类。照片侧有世界照片的坐标、日期、缩略图和高清图。
这些数据原本都长得不一样。电影有导演、国家、评分和观影日期。书有作者、出版信息、故事地，或者作者所在地。音乐有城市、歌手、曲目和流派。照片有经纬度、时间和图像资源。
Pocket Earth 做的第一件事，就是把它们整理成地图能理解的统一结构。
静态数据会进入 MAP_MARKERS。运行时新增的数据会进入 userMarks、planets 和 geoStickers。然后通过 toGeoJSON 统一转成 GeoJSON，再交给 Mapbox 渲染。
这里的重点不在数据量。更关键的是，电影、书、照片、音乐、行程、心情这些不同类型的个人对象，最后都能收敛成同一种空间对象。
这样一来，地球就成了统一索引。无论来源是电影、书、照片、歌单，还是某个 agent 刚生成的结果，最后都能进入同一套地图图层。
第二个关键点，是实时联动。
这颗地球是活的。它会订阅运行时的数据变化。
比如 mood-curator 写入一句心情。
比如 planet-builder 生成一颗日落星球。
比如 travel-curator 完成一段行程，把停留点钉到地球上。
这些动作会写入对应的 localStorage store，然后出现在地图上，变成一个位置、一个图层、一张贴纸，或者一个可以点击查看的详情。
具体链路是这样的。
agent 产生一个 mark。
这个 mark 写入 userMarks、planets 或 geoStickers。
store 发出订阅通知。
MyMapTab 合并静态标记和运行时标记。
最后调用 Mapbox 的 source.setData。
地图图层即时重绘。
这段链路对我很重要。它说明 Pocket Earth 里的 agent 答完一句话以后，结果还会继续进入系统状态，落到地球上的某个地方。
所以在这个项目里，地球是系统的一等公民。它承担的是所有 agent 输出的汇合点。
第三层是 frost-agent。
frost-agent 是一套多 agent harness。可以把它理解成一个分工明确的协作框架。
最外面是 Shell。Shell 负责对外统一声音。用户感受到的是同一个系统在说话，背后换了不同 agent，语气也不会突然变成另一套。
下面是 Router。Router 负责判断用户这句话应该交给谁。明确命令先用规则快速处理。能够在端侧分类的，就先走端侧 classify。复杂、少见、上下文更长的问题，再交给云端 Brain 判断意图。如果云端不可用，就回到规则兜底。
再下面是 Brain。Brain 是云端模型，负责写作、解释、推荐、串词和复杂回答。项目里通过 /api/frost-llm 走服务端代理，DeepSeek 的 key 不会进入前端。
同时还有 Edge Selector。它是端侧模型层，负责 classify、rank、embed、vision 和轻量 chat。它的目标很明确：完成挑选、排序、匹配、识别这些高频任务。
Memory 和 Profile 负责记忆。Memory 管会话内最近几轮，Profile 管跨会话的长期口味画像，比如导演、作者、流派、城市这些偏好标签。
最后是 Boundary。Boundary 是动作落地前的校验层。子 agent 只产生动作建议，真正写入地图之前，必须先过 validator。
这套结构可以简单理解为：Router 决定交给谁，端侧 Selector 负责挑和找，云端 Brain 负责写和解释，Boundary 判断能不能落地，地球负责把结果显形。
我最想重点讲的是端侧模型。
这个项目没有把所有事情都扔给一个云端大模型。它把模型能力拆成了两类：端侧 Selector 和云端 Brain。
端侧 Selector 处理的是高频、隐私敏感、输入很多但输出很少的任务。
比如照片整理。相册里可能有几千张照片。如果全部上传给云端模型，从隐私、成本和速度上都不理想。photos-curator 的理想形态，是端侧视觉模型先看图，打标签，给收藏价值打分，再把少数高价值结果交给后续链路。
比如音乐推荐。曲库里可能有几百首候选歌。端侧可以先根据场景和口味做 rank，把候选缩小到一小撮。云端 Brain 再负责精选、写推荐理由和串词。
比如旅行规划。目的地 POI 很多，用户偏好也比较私人。端侧可以先按喜好排序，再把结构化结果展示出来，或者交给云端写成更自然的行程说明。
对话层也一样。用户输入一句话，端侧可以先 classify 意图。如果是明确类型，就不用每次都消耗云端模型。
未来这条端侧链路接的是 MNN × Qwen。
文本模型是 Qwen3.5-0.8B。视觉模型是 Qwen3-VL-2B。在 Apple Silicon 上可以利用 Arm SME2 加速。
代码里用 EdgeModel 契约把这层包起来。前端调用的是 edgeSafe.classify、edgeSafe.rank、edgeSafe.vision 这些稳定方法。后面到底接 MNN、ollama 还是 stub，调用方都不用关心。
如果端侧后端不可用，系统也不会崩。edgeSafe 会按 mnn、ollama、stub 做三级降级，并记录 health。端侧能力是增强项，系统不会因为它暂时缺席就停摆。
端侧模型的意义很直接：隐私数据尽量留在设备上，高频筛选尽量少上云，云端只处理筛过的小输入。这样既省 token，也更稳定。
云端 Brain 的职责刚好互补。
它负责质量敏感的写作和表达。
比如歌单推荐理由。比如一座城市的文化解释。比如读书 agent 根据我的书架做主题推荐。比如电影 agent 根据我的豆瓣记录判断我可能喜欢什么。比如 council 里多个 agent 的发言。
这些任务需要语言质量、联想能力和叙事能力，所以更适合交给云端 Brain。
但云端 Brain 接收到的输入，应该已经整理过。我希望它看到的是端侧筛选后的结果、结构化摘要，以及压缩过的长期画像，避免一股脑吞下所有原始数据。
这也是 provider-compat 的作用。模型请求格式不应该散落在业务代码里。DeepSeek、Qwen，或者以后别的模型，都应该被 adapter 收敛到统一入口。
这样以后换模型，不需要重写业务逻辑，只需要加一个新的 provider adapter。
这里可以讲一个更适合这个项目的真实例子：音乐电台。
Pocket Earth 里有一个 music-curator。我觉得它是最能说明端云分工的 agent。
它不用我手动选一堆歌，再做成歌单。它的机制是：我说一句此刻的状态，系统自己把一份歌单建出来。
比如我说：“我正在看海明威，帮我建立下歌单。”
这句话先进入 Router。Router 判断这是一个 open-dj 类型的请求，也就是要开启一段对话驱动的电台。
系统会先理解“海明威”这个场景：克制、硬朗、海、酒吧、战争后的疲惫，还有那种话很少但情绪很重的气质。
接下来，系统也不会直接把整个曲库丢给云端大模型。曲库里有几百首歌，如果每次都让云端从头读一遍，成本高，速度慢，也没有必要。
所以第一步交给端侧 Selector。
端侧做的事情很朴素：挑和排。它根据“海明威”“阅读”“克制”“海边”“夜晚”这些线索，在本地曲库里先做 rank，把几百首候选缩小成一小撮更可能贴合的歌。
然后才轮到云端 Brain。
云端 Brain 负责写。它从端侧筛出来的候选里精选几首，给每一首写推荐理由，解释为什么这首歌适合此刻读海明威。
同样，如果我说：“我正在看波拉尼奥，帮我建立下歌单。”
它理解到的就会换成另一种气质：流亡、夜行、诗人、侦探、拉丁美洲、漫游，还有一点危险和狂热。
端侧先把候选歌按这个气质重新排序，云端再写出更贴近波拉尼奥的推荐理由。
所以这里真正发生的事情，其实是一句话变成了一个电台任务。
Router 判断意图。
端侧 Selector 先挑歌。
云端 Brain 写串词。
music-curator 生成播放队列。
最后歌曲和城市一起回到地球上。
这份结果会变成一个真正可播放的电台队列。每首歌有自己的音频、城市封面、DJ 解说词、DJ 解说音频，也会落到歌手城市或者歌曲相关城市。
也就是说，这里完整跑了一遍 Pocket Earth 的核心链路。
我说一句话，系统理解意图，端侧先筛，云端再写，curator 生成可播放的结果，地球把歌曲和城市显出来。
这个例子比普通推荐更有意思的地方在于，模型给出的结果已经进入了界面、播放队列和声音层次。
歌单变成了电台。
电台里还有一个沉浸式播放器。中间是城市封面，下面是播放进度、上一城、下一城、播放按钮，还有 DJ 开关。
如果打开 DJ，Frost 会开始介绍当前这首歌。介绍音频播放时，主音乐会自动压低，DJ 声音在前面，音乐变成背景。讲完之后，音乐音量再恢复。
这就是一个很小但完整的 agent 产品闭环。它会改变界面，改变播放队列，也改变声音的层次。
还有另一种电台机制，叫 radio-24h-director。
它甚至不需要云端大模型。它是纯本地确定性的。
它会读取所有电台城市的时区，计算每座城市的当地时间，找出此刻哪座城市最接近日落，比如当地 18:30。然后沿着日落线，把接下来即将进入日落的城市排成一个顺序。
所以用户说“跟着日落走”，系统就能生成一条从现在到午夜的城市电台路线。
这件事特别符合 Pocket Earth 的气质。
因为音乐在这里也被地理化了。每一首歌属于一个城市，每一座城有自己的时间，每一次切歌都像是地球转到了下一个地方。
这就是我想要的效果：端侧负责挑，云端负责写，纯规则负责时间编排，地球把这一切显出来。
然后是 curator。
这里没有让一个大模型什么都聊。每个 curator 都有自己的职责边界。
photos-curator 是执行型。它面对的是高噪声输入。几千张照片进去，最后只应该输出少数高价值照片、重复清理建议，以及它们在地球上的落点。这个场景特别适合子 agent，因为输入远远大于输出。
music-curator 是流水线型。用户说一个场景，比如“我正在看海明威，帮我建立下歌单”，或者“我正在看波拉尼奥，帮我建立下歌单”。系统先理解场景，再由端侧排序候选曲，再由云端精选和写理由，最后把歌曲或者歌手城市钉回地球。
books-curator 和 movies-curator 是数据接地型。左边是真实书架和观影记录，右边是对话 agent。agent 回答时会注入这些数据，所以它的推荐和解释会基于我的记录，避免泛泛而谈。
planet-builder 和 mood-curator 是运行时写入型。一个可以根据一句话生成主题星球。一个可以把一句心情解析成地点，再写回地球。
council 是团队型。多个 agent 可以围绕同一个议题发言。但它不会无限互聊。它有固定轮次，有发言顺序，也支持 AbortSignal 随时喊停。
这样拆的好处是上下文隔离。
每个 curator 只处理自己的输入、工具和输出。主 agent 不需要吞下所有原始数据，只需要接收结构化结论。这样既能保护主对话的上下文，也能让权限和边界更清楚。
最后说 Boundary 和 Profile。
我不希望子 agent 直接改全局状态。所以子 agent 输出的是动作建议。真正写入之前，要先经过 validator。
比如地点是否合法。
比如照片分数是否过阈值。
比如同一个对象、同一个地点是否重复。
这些都应该在 Boundary 里挡住。
Profile 也类似。它不保存原始隐私文本，只保存偏好标签计数。比如常看的导演、常读的作者、常听的流派、常出现的城市。
并且它有 fingerprint 缓存。如果口味画像没有实质变化，就不重复调用云端 Brain 总结。这样既省成本，也避免无意义的重复生成。
所以这套系统的边界很清楚：端侧先筛，云端再写，子 agent 只提建议，Boundary 负责提交，Profile 只存压缩后的偏好标签。
最后说一下为什么叫 Frost。
这个名字来自《趁生命气息逗留》里那台想理解人类的机器。小说里的 Frost 会委派漫游机，在地球上收集书、电影、音乐和文物，试着从碎片里理解人类。
这个项目里的 frost-agent 也有类似的气质。主 agent 不亲自吞下所有数据，它会委派一个个 curator，把我生活里的碎片整理好，再钉回地球。
它收集碎片，是为了理解“人”。
我收集自己的碎片，是为了在这颗口袋地球上，拼出“我”。
所以 Pocket Earth 最后呈现出来的，是一套 agent harness 的可视化实验。
数据先归一成地理对象。
地球作为统一索引。
端侧模型负责挑和找。
云端模型负责写和解释。
Boundary 负责判断能不能落地。
最后，所有结果都回到同一颗地球。
端侧管挑和找。云端管写。地球是统一索引。
Pocket Earth 想做的，就是把我的世界，钉回它该在的地方。
