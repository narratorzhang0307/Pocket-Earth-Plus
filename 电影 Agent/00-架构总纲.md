# 电影整理 Agent · 架构总纲

> 与「照片整理 agent」同源的六层解耦架构（感知/记忆/推理/行动/反思/协作），但输入从「像素」换成「一个片名 / 一张截图 / 一句话」，知识从「EXIF + 端侧 CLIP」换成「本地豆瓣库 + 云脑补全」。
> 解耦原则同照片：`src/app/lib/movie/` 自成一体，不动 FROST 内核（FrostIntent 封闭枚举不碰），只当 userMarks 总线的生产者。

## 0. 一句话定位

用户给一个**片名 / 一句话「我看了 xx」/ 一张电影截图**，agent 自动解析→补全成一张**带多维标签的电影票根**（导演 / 演员 / 类型 / 流派 / 剧情 / 我的评分），并由地理子 agent 判出**取景地 / 故事地**，建议钉到地球。全程 suggest-then-confirm，用户确认才落地。

## 1. 核心数据契约（verdict ⟂ pin 的电影版）

两条正交输出，沿用照片 agent 的「判定与钉点解耦」：

- **MovieTags**（内容标签，决定票根长什么样）：`director / cast[] / genre(类型) / movement(流派) / plot(剧情一句) / userRating(我的评分)`
- **GeoTarget**（钉点，决定钉哪、能不能钉）：`kind: 'filming'(取景地) | 'story'(故事地) | 'country'(国家兜底)` + `place / lng / lat / confidence`

标签齐不齐 ≠ 能不能钉；能不能钉只看 GeoTarget 解析出没解析出坐标。

## 2. 六层职责（对标照片 agent）

| 层 | 文件 | 职责 | 对应照片 agent |
|----|------|------|----------------|
| 感知 | `sense.ts` | 三种输入归一成「候选片名 + 可选评分」：①一句话抽片名（《》优先 / 去「看了·标记·帮我·这部电影」噪声 / 抽「8分·五星」）②截图 OCR（端侧 vision，原图不出端）③手填 | `features.ts`（EXIF/像素）+`vision.ts`（CLIP） |
| 记忆·知识 | `catalog.ts` | 本地豆瓣库当 RAG 知识底座：归一化片名模糊匹配 → 确定性元数据（导演/国家/年份/豆瓣分/简介），离线即得 | `store.ts` 的已知索引 |
| 推理 | `resolve.ts` | **兜底解析链**：本地库命中→云脑补全→手填骨架，逐级降级、单级失败不抛错（舱壁） | `reasoning.ts` 的 classify/decide |
| 协作·子agent | `tagging.ts` | 两个运行时子 agent：①**补全子 agent**（云脑 JSON：导演/演员/类型/流派/剧情）②**地理子 agent**（取景地→故事地→国家，逐级 geocode） | dedupAndCluster + 多信号 |
| 反思 | `critic.ts` | 确定性护栏：评分钳 0-10、年份合理性、坐标合法、幻觉演员剔除、对同一片幂等去重、应用用户历史纠错 | `critic.ts`（同名同职） |
| 行动 | `pin.ts` | suggest-then-confirm：draft 仅建议，用户确认才 `addUserMark(kind:'movie')`，写全标签 meta + 喂长期画像，绝不自动落 | `geoPin.ts` |
| 编排 | `agent.ts` | 串流水线 sense→resolve→tag→geo→critic→draft；阶段 onPhase 回调；`confirmPin(draft)` 落地 | `screen.ts` |

## 3. 兜底策略链（用户两种说法 + 一种截图）

```
输入
 ├─ 一句话「我看了《xx》/ 帮我标记 xx」──┐
 ├─ 截图（电影海报/详情页）── OCR 出片名 ─┤→ 候选片名
 └─ 手填（片名+国家+年份+星级）──────────┘
            │
   ① 本地豆瓣库模糊匹配  ── 命中 → 拿到 导演/国家/年份/豆瓣分/简介（确定锚点）
            │ 未命中 / 缺字段
   ② 云脑补全子 agent（/api/frost-llm json:true）── 出 演员/类型/流派/剧情 + 取景地/故事地 + 补 国家/年份
            │ 云脑不可用
   ③ 手填骨架（needsConfirm=true，只有片名+用户星级）
            │
   ④ 地理子 agent：取景地→故事地→国家，逐级 geocodeCity / movieCountry
            │ 都没有
   ⑤ 「无法定位」→ 票根仍可存，标 needPlace，等用户手选国家
```

每一级都是**确定性优先、概率兜底**：本地库（确定）→ 云脑（概率）→ 用户（最终裁决）。对标黄佳《Harness 工程之道》的「按难度分模型 + 单条舱壁降级」。

## 4. 子 agent 设计（协作层）

- **补全子 agent**：一次云脑调用，强约束 JSON schema（缺字段留空字符串/空数组，禁编造演员）。系统提示明确「不确定就留空，不要编」——把幻觉风险压在护栏前。
- **地理子 agent**：纯端上，不联网。取景地名 / 故事地名 → `geocodeCity`（共享城市字典）；落空 → `movieCountry(country)`（国家代表城市）；再落空 → needPlace。返回 `GeoTarget{kind}`，UI 显示「钉在 取景地·xx / 故事地·xx / 国家·xx」让用户一眼知道精度。

## 5. 隔离与隐私

- 截图原图不出端：仅走 `edgeSafe.vision`（端侧），与照片 agent 同纪律。
- 本地索引（IndexedDB `pe-movies`）只存派生小结论（归一片名→标签），不存截图。
- 云脑只收「片名 + 少量提示」，不收用户全片库。

## 6. 与已有设施的接线

- 写落点：`addUserMark({kind:'movie', meta})` → 地球层已订阅、`resolveDetail` 已认 movie（扩展 meta 显示 演员/类型/流派/剧情）。
- 运行页：增强现有 `MoviesRunPage` 的「记一笔」——输入/截图后先跑 agent 出 **draft 预览卡**（全标签 + 取景地/故事地），用户确认再钉；保留纯手填兜底。
- 长期画像：`recordSignals('movies', {countries,...})` 增量喂 profile（沿用现状）。

## 7. 不做 / 边界

- 不碰 FrostIntent 枚举、不改 router 内核（P2，照搬照片 agent 的决定）。
- 不接第三方电影 API（无 key、不稳）；知识来源=本地豆瓣库 + 云脑常识。
- 不自动钉、不自动删：一切 suggest-then-confirm。
