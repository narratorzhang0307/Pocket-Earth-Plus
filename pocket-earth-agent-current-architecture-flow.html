运行链路

  1. 用户在 MusicAgentRunPage、通用 AgentChat，或各 curator 运行页发起操作。
  2. 音乐对话层调用 runFrost(ctx)，传入当前时间、用户文本和最近对话。
  3. switch-handler 先做明确指令匹配：下一首、暂停、切城等不动用模型。
  4. 不是明确指令时，Router 调 edgeSafe.classify 做端侧预分类；命中合法意图就直接 dispatch。
  5. 端侧没把握时，llmRoute 走云脑 /api/frost-llm 判意图、抽实体；云不可用则正则兜底。
  6. 意图通过 intentRegistry 找到处理器，委派到 runtime agent 或页面内 curator 逻辑。
  7. 子 agent 返回 Frost 回复、结构化数据、trace，以及可选动作建议。
  8. validator.ts 只放行已注册动作；未来 mark_place 也按同一模式接校验器。
  9. 播放动作进入播放器；地理落点进入 userMarks / planets / geoStickers 并触发地图刷新。
  10. Memory 负责最近对话，Profile 负责长期口味摘要；Health 记录 edge/cloud 各步骤成败，让降级可见。

为什么是 Harness

主 Frost 不吞原始噪声，只委派 curator，并只接收结构化结论。照片、书影、网页补全这类高噪声任务被隔离在子上下文里。

为什么是地球

地理坐标是所有对象的共同索引。书、电影、音乐、照片、行程与心情不再只是列表，而是回到“在哪里”。

为什么端侧优先

个人偏好、相册原图、截图识别、排序打分都尽量在端上完成；云端只接收最小上下文做高质量生成。
