# 技术说明：端侧的必要性，与一种用端侧能力构建 skill 的方式

> 以「端侧识图→结构化 skill（visionExtract）」为具体案例。可直接用于方案/可行性说明文档。

## 一、端侧到底解决了什么——一个绕不开端侧的真实场景

应用要做的事：把用户随手的一张图（一张电影票根、一本书的版权页、一张观鸟记录、一张咖啡馆小票），自动读成结构化信息并落到地图上。

要把图变成数据，第一步绕不开「读图」。而用户的图里几乎总混着敏感信息——票据上的证件号、订单里的手机号、截图角落的定位、二维码。**只要"读图"这一步发生在云端，就意味着把用户的原图整张交出去。**

我们的处理把边界划得很清楚：

```
一张图 ──① 端侧视觉模型读图──▶ 文本 ──正则脱敏──▶ ② 结构化(端侧优先/可回退云)──▶ 结构化 JSON
        原图只在这一步、且只进端侧            敏感号 ***          原图不参与第②步
        【原图一个字节都不出设备】
```

- **第①步读图：只在端侧（Qwen-VL，经 MNN / 设备 GPU）**。原图不出端。
- **第②步结构化：只拿端侧读出、且脱敏后的纯文本**，可在端侧做，也可按需交云端模型——但**原图从不参与第②步**。
- **端侧不可用时诚实降级**：不偷偷把原图送云，而是请用户手填。降级本身也守住了隐私。

**结论：这是一项云替代不了的能力。** 不是"端侧更快更省"这种优化级理由，而是"没有端侧就做不到（做了就泄露隐私）"这种必要级理由。端侧在这里同时换来三件事：隐私（原图不出端）、离线（图就在设备上）、低延迟（不传大图、不等网络往返）。这正是"核心交互逻辑本地运行 + 云端协同"的标准形态：核心与隐私相关的部分在端，云只处理脱敏后的文本。

## 二、一种用端侧能力构建 skill 的方式（四条做法）

我们把这类能力沉淀成「skill」——一个职责单一、可被多处调用、可单独打包迁移的模块。构建时遵循四条：

1. **能力做成可注入的接口，而非写死的调用。**
   skill 不直接绑定某个端侧框架，而是依赖两个注入进来的接口：`visionOnDevice`（端侧视觉）与 `structure`（文本结构化）。把接口换成任意实现（MNN、设备 GPU、不同的端侧/云模型），skill 整体即可迁移。这让"端侧"成为一个可替换的后端，而不是硬编码——也是它能单独打包、给第三方直接读懂的原因。

2. **领域差异做成参数，而非分支。**
   电影有导演、书有作者、观鸟有物种——字段各不相同。我们不为每个领域各写一套识图逻辑，而是把"目标字段表"做成调用方传入的参数。一套 skill 适配任意领域，新增领域零改 skill。用户甚至能自己造一个新的"整理器"，声明几个字段，就立刻拥有"拍图入库"能力。

3. **隐私写进数据流，而非写进说明。**
   "原图不出端"不是一句承诺，而是代码结构保证的：唯一接触原图的就是端侧那一个接口；脱敏用确定性正则在数据离开端侧之前完成，不依赖模型自觉。

4. **skill 之间可组合，把隐私边界收口到一处。**
   我们把它拆成图↔文对称的三个 skill：`visionRead`（图→脱敏文本）、`textExtract`（文本→字段）、
   `visionExtract`（图→字段）= `visionRead` + `textExtract`。
   "原图只进端侧"这条线、以及脱敏正则，都**只在 `visionRead` 一处**——上层和所有 agent 都只跟脱敏后的文本打交道。
   这样图片输入走 `visionExtract`、纯文本输入走 `textExtract`，两条路对称、共用同一套字段契约：skill 调 skill、能力像积木一样拼起来。

## 三、这套做法带来的工程收益

- **端侧成为核心机制**：端侧不是"云的降配备份"，而是承担隐私与核心交互的主力；云只处理脱敏文本。
- **泛化**：同一套 skill 被电影、书、旅行、用户自建的任意"整理器"共享，一处实现、处处可调。
- **可移植 / 可解耦**：因为只依赖接口，这套 skill 能从应用里整体取出、单独成包，在别的项目、别的端侧框架上直接落地。

## 四、已落地的证据（不是设想，是已上线的结构）

在我们的应用里，多个 agent 已经各自做好接口、接入同一套 skill：

| agent | 接入哪层 | 说明 |
|---|---|---|
| 电影认片 / 书认书 | `visionExtract` | 传单字段 schema（片名 / 书名），取结构化结果 |
| 用户自建 agent | `visionExtract` | 声明几个字段就白得"拍图入库"，零改 skill |
| 旅行票据整理 | `visionRead` | 行程是嵌套结构（多段行程 / 多段住宿 / 多个景点），故取纯文本后自行结构化 |

各 agent 此前各自直连端侧视觉、各有一份脱敏正则；接入后，"原图只进端侧"这条线与脱敏逻辑统一收口到底层 `visionRead` 一处。新增一个领域 = 传一份新 schema，不改 skill。

附：可运行的独立实现见同目录 `visionRead.ts`（图→文）、`textExtract.ts`（文→字段）、`visionExtract.ts`（图→字段=前两者组合），四种场景的接入示例见 `example.ts`，接口/架构/推荐模型见 `README.md`。
