# Pocket Earth · 技术难点与解决方案

## 总览

Pocket Earth 是一个跑在浏览器（iOS PWA、无原生层）里的端云双脑多智能体系统：用户随手记一笔、丢一张截图、读完一本书，系统就把它钉到一颗可探索的地球上。它在工程上啃下的硬骨头，几乎都不在「让模型更聪明」这一侧，而在**编排与工程**。

模型清单（全程只用通义 Qwen 系）：

- **云脑**：通义 qwen-plus（DashScope，OpenAI 兼容接口），负责生成 / 润色 / 长尾意图判读。
- **端侧（真不出端）**：浏览器内 WebLLM 跑 **Qwen3-0.6B**（WebGPU、动态 import、不进主 bundle）。
- **端侧（服务端代跑）**：本机 demo 用 ollama 跑 **qwen3:0.6b**（文本）+ **qwen2.5vl:3b**（视觉）；手机 / 生产用 MNN 部署 **Qwen3-0.6B + Qwen2.5-VL-3B**。
- **stub**：无任何模型时的规则兜底（返回空 / 均匀分），调用方自动降级。

下面是它的几条工程主线：

- **端云协作与优雅降级**——无云脑（没配通义 DashScope）、无端侧（没启用浏览器内模型）、无网络（PWA 离线）三种缺位都能跑，缺位是「静默降一级」而非「崩」；端侧后端（浏览器内 Qwen3-0.6B ⇄ 服务端 ollama/MNN ⇄ stub）可热切换而调用点零改动；前端不持任何密钥也能用云脑。
- **鲁棒性：真实故障的兜底**——截图认片卡死 176s 的异步无超时、明说电影却落心情的判域盲区、同名异地钉错洲、触屏删不掉、持续部署白屏、瞬时故障被一次性吞掉……每一条都有真实 commit 佐证。
- **端侧性能**——整理相册单张从 120s+ 压到 1~2s，批处理不 OOM，长期画像口味未变就不请云脑。
- **长期记忆与隐私边界**——回流闭环、双层记忆解耦、原图/画像物理不出端的双向隐私边界。
- **编排泛化与可观测**——注册表化内核、确定性护栏、约 40 行的 FrostBus 事件树、学习型技能的安全闸。

下面按主题分组，每个难点用统一格式陈述：**问题 → 为什么难 → 解决方案 → 代码佐证 → 效果**。

## 一、端云协作与优雅降级

### 1.1 「无云脑 / 无端侧 / 无网络」三种缺位都能跑——把优雅降级做成第一公民

**问题**：多智能体 harness 默认假设 LLM 永远在；但比赛/演示现场常缺：服务端没配 DashScope key（无云脑）、用户没启用浏览器内模型且服务器没装 ollama（无端侧）、PWA 离线（无网络）。

**为什么难**：任一缺位若直接抛错或返回 null，会沿调用链炸穿整个 agent 流水线（钉地球/记一笔/法庭全挂）。难点是让缺位变成「静默降一级继续」，且不能让每个调用点各写 try-catch。

**解决方案**：双脑各设一个「安全空值即触发兜底」的契约面。云脑侧：`stubBrain.complete()` 默认返回空串，`httpBrain` 在 `!r.ok` 或 catch 时也返回空串；各子 agent 约定「检测到空串就走自己的规则 fallback」。端侧：`contract.ts` 的 `withFallback()` 给 `EdgeModel` 每个能力包 guard，异常一律返回该能力的安全默认（`available→false`、`chat/classify/vision→''`、`rank/embed→[]`），`httpEdge` 在 `!r.ok`/超时/catch 时返回 `{backend:'stub'}`。服务端 `edgeStub()` 再给 rank/classify/embed 造确定性占位（rank 按原序线性给分、classify 取首个 label）。三层空值层层兜住，最终所有 agent 在零 LLM 下退化为规则版仍可钉点。

**代码佐证**：
- `frost-agent/harness/brain.ts:6-11`（stubBrain 空串）
- `frost-agent/harness/httpBrain.ts:13-18`（`!r.ok`/catch→`''`）
- `frost-agent/edge/contract.ts:15-28`（withFallback guard + 安全默认）
- `frost-agent/edge/httpEdge.ts:20-27`（超时/失败→`{backend:'stub'}`）
- `server.mjs:192-198`（edgeStub 确定性占位）

**效果**：任一后端缺位都只是「能力降级」而非「功能崩溃」：无 key 时 agent 走规则版照常钉地球，端侧不可用时退服务端再退 stub，全链路无一处需要调用方写 try-catch。降级路径还经 `recordHealth` 记成可观测的步骤健康（`health.ts`），便于 UI 提示「已降级」。

### 1.2 端侧后端可热切换（浏览器内 Qwen ⇄ 服务端 ollama/MNN ⇄ stub）而调用点零改动

**问题**：端侧推理有三套互不兼容的后端：浏览器内 WebGPU 跑的 **Qwen3-0.6B**（真不出端，但需用户点启用 + WebGPU 支持）、服务端 ollama（文本 qwen3:0.6b、视觉 qwen2.5vl:3b）、生产 MNN sidecar（部署 Qwen3-0.6B / Qwen2.5-VL-3B）。

**为什么难**：它们的 API 形态、就绪时机、可用能力都不同（文本三件套都能，embed/vision 浏览器内不做）。若让每个 agent/router 自己判断「现在该调哪个后端」，切换逻辑会散落几十处，加一个后端要改全仓。

**解决方案**：在 `contract.ts` 立一个对调用点透明的 `routed:EdgeModel` 路由层：文本三件套（chat/classify/rank）按 `isWebllmReady()` 动态择路——浏览器内 Qwen 就绪走 `webllmEdge`（真端侧、数据不出端），否则回退 `httpEdge`（服务端 `/api/edge`）；embed/vision 浏览器内不做，始终走 `httpEdge`。整条路由再被 `withFallback` 包成单一出口 `edgeSafe`。`webllmEdge` 用动态 import 把 web-llm 挡在主 bundle 外（用户点启用才下 ~400MB 权重），WebGPU 不支持时 `available()=false` 自动回退。换端侧模型只改 viteEdge 实现与本契约，agent/router 不动。

**代码佐证**：
- `frost-agent/edge/contract.ts:33-43`（routed 按 isWebllmReady 双路由 + edgeSafe 单一出口）
- `frost-agent/edge/webllmEdge.ts:1-8,52-79`（动态 import 不进主 bundle、WebGPU 不支持→回退）
- `server.mjs:180-186`（pickBackend 服务端侧 mnn→ollama→null 再选）

**效果**：调用点统一只认 `edgeSafe`，端侧后端从「浏览器内 Qwen」切到「服务端 ollama/MNN」再切到「stub」全程无感知；真端侧路径数据不出浏览器（满足「核心交互逻辑本地运行」），且 web-llm 不污染主 bundle，未启用端侧的用户零额外下载。

### 1.3 前端不持任何密钥也能用云脑——服务端代理 + dev/prod 双路镜像

**问题**：PWA 前端 bundle 是公开可下载的，任何写进 bundle 的 DashScope key 等于公开泄漏；但 agent 又必须能调通义 Qwen。

**为什么难**：还有个隐藏陷阱——dev（vite 中间件）和 prod（server.mjs）是两套独立代码，若行为不一致（比如一边吞掉上游 5xx），客户端重试逻辑会在某一环境失效。

**解决方案**：前端只 POST 到同源 `/api/frost-llm`，密钥从不在前端：`server.mjs` 从 `process.env.DASHSCOPE_API_KEY` 读、`vite.config.ts` 从 `loadEnv` 读，key 只在服务端进出站请求的 Authorization 头里拼。无 key 时服务端返 `{text:'',error:'no_key'}`，前端拿到空串即走规则兜底。dev 中间件与 prod 路由刻意镜像同一语义，且都把上游 429/5xx **透传**（直接写 `r.status` 而非恒 200），让客户端 `enrichJSON` 的 `withRetry` 能据 `r.ok` 区分「瞬时故障该重试」与「恒空串重试形同虚设」。`main.tsx` 一行 `setFrostBrain(httpBrain)` 完成注入，子 agent 不感知后端是真 LLM 还是 stub。

**代码佐证**：
- `server.mjs:33-37`（key 从 env 读、无 key 则 LLM=null）
- `server.mjs:66`（无 key→`{error:'no_key'}`）
- `server.mjs:71-73,83`（Authorization 头服务端拼、429/5xx 透传）
- `vite.config.ts:11-14,31,38`（dev 中间件镜像）
- `src/main.tsx:8-9`（一行注入 httpBrain）

**效果**：公开 PWA bundle 不含任何明文密钥即可用云脑，密钥泄漏面收敛到服务端 `.env`；dev 与 prod 行为一致，上游瞬时抖动（429/5xx）被透传给客户端 `withRetry` 重试而非被一次性吞成空串白丢；无 key 部署也不崩，直接退规则版。

## 二、鲁棒性：真实故障的兜底

### 2.1 端侧识图全链路无超时 → 截图认片卡死 176s：双层纵深超时兜底

**问题**：截图认片走 `visionRead → edgeSafe.vision → httpEdge.call()` 的 `fetch('/api/edge')` → 服务端 ollama/MNN 的视觉模型（ollama qwen2.5vl:3b / MNN Qwen2.5-VL-3B）推理，整条异步链路**没有任何一处设超时**。

**为什么难**：原生 `fetch` 本身无默认超时，大截图叠加 VL 模型冷加载/推理一旦挂起，fetch 不 reject、只是永不 resolve，层层 await 永久 pending。这类故障最阴险——它不是抛异常（try/catch 兜得住），而是「永不返回」，任何 catch 都拦不到一个永不到来的 reject，UI 只能无限转圈（实测卡死 176s）。这是异步控制流的根因 bug，极易在 demo 现场暴露。

**解决方案**：双层纵深防御，且两层都避开热区文件：① 治本层 `httpEdge.ts` 的 `call()` 用 `AbortController` 按 task 分档超时（vision 35s / chat 20s / 其余文本小模型 15s），超时 `ctrl.abort()` → fetch 抛 AbortError 落 catch → 返回 `{backend:'stub'}`，真正释放在途连接；② 收口层 `visionRead.ts` 唯一出口再兜一道 `Promise.race(edgeSafe.vision vs setTimeout(30s)→'')`，超时返回空串 → 上层 sense 拿不到片名 → 走「没认出·去手填」兜底。两层互为纵深：race 保证 UI 一定脱困、AbortController 保证连接真释放，且 race 还覆盖了 `book/notes.ts` 等绕过 visionRead 直调 `edgeSafe.vision` 的路径。

**代码佐证**：
- commit `0a4f183`
- `src/app/lib/skills/visionRead.ts:38-46`（`Promise.race` + `setTimeout(timeoutMs??30000,'')`）
- `frost-agent/edge/httpEdge.ts:7-26`（AbortController + 按 `body.task` 分档 35000/20000/15000 + finally clearTimeout）

**效果**：截图认片不再可能无限转圈：无论端侧 VL 冷加载多久、推理是否挂起，最迟 30-35s 必脱困并落到手填兜底；同时 abort 真正释放挂起的 TCP 连接，避免连接堆积。覆盖电影/书/旅行/自建 agent 全部经 visionRead 的截图认图路径。

### 2.2 有截图 + 有文字时判域只喂文字、片名困在截图里 → 明说电影却落心情：确定性判域词护栏

**问题**：`route.ts` 的分支 `imageDataUrl && !t` 为假时（既有图又有字）会走 `classify(t)`，而 `classify` 只把**文字**喂云脑、根本不传截图。用户文案「帮我记录下这部电影到我的电影 agent 中」里没有具体片名（片名只存在于截图像素里），云脑只看到一句没有作品名的话 → 按既有规则「没作品名→回落 mood」把一部电影判成了心情贴。

**为什么难**：这不是模型不够聪明，而是**喂给模型的输入天然缺片名**，纯靠加强 prompt 治不了，必须用确定性规则在判域结果上做硬纠偏。已有的 travel/想去护栏管不到这条路径。

**解决方案**：在 `classify` 之后、travel 护栏之前插入一道**确定性判域词护栏**：文字含「这部电影/影片/剧集/这部小说/这本书」等明确表域的词组时，强制 `domain=movie` 或 `book`，把截图里的片名交给对应 agent 的 vision 去认。关键纪律是正则**收紧为词组而非裸词**——只匹配「这部电影」「这本书」，不匹配裸「电影」「书」「作者」，从而避免「电影博物馆」「软件作者」这类被误判。

**代码佐证**：
- commit `f744fe1`
- `src/app/lib/capture/route.ts:77-79`（`/这部(电影|片|剧|影片|纪录片)|这部片子|影片|剧集|看的这部/i → 'movie'`；`/这本(书|小说)|这部小说|看的这本/i → 'book'`）

**效果**：9 例自检全判对：「帮我记录下这部电影」+ 截图不再误落心情、改判 movie 走 vision 认片名；同时「电影博物馆/想去/去了京都/城市与狗」等不误伤（仍交 classify）。配合同轮 vision 超时（`0a4f183`），截图认片既不卡死、明说域也不再误落心情。

### 2.3 同名异地：中文裸地名被 Mapbox 匹配到错误国家 → 钉错洲：拼国家上下文消歧

**问题**：`resolvePlace` 调 Mapbox geocoding 时 `limit=1`（只取第一个匹配）且无国家上下文。当 book/movie agent 让云脑补出的故事地/取景地是中文裸地名时，歧义全爆：用户读完略萨《城市与狗》、云脑正确补出故事地「利马」（秘鲁首都），但光秃秃的「利马」喂 Mapbox 被匹配到塞浦路斯利马索尔 / 委内瑞拉 Lima Blanco → 钉错洲。

**为什么难**：这是全球地名消歧的经典坑（同「圣地亚哥」同名问题）。模型那一层是对的（确实给出了正确城市），错在地理编码这一层把对的城市名解析到了错的坐标，且 `limit=1` 没有给二次纠偏的余地。

**解决方案**：在 `book/tagging.ts` 与 `movie/tagging.ts` 的 `geoResolve` 里（两文件镜像同改），调 `resolvePlace` 时若云脑已补出 `country` 字段，就把地名拼成「地名 国家」（如「利马 秘鲁」）再查 → Mapbox 拿到国家上下文后钉对秘鲁利马 `[-77,-12]`。故事地、作者地/取景地三级落坐标都加上这层消歧。命中本地字典的（东京等中文子串匹配）不受影响。

**代码佐证**：
- commit `7a4a4dd`
- `src/app/lib/book/tagging.ts:59-62` 与 `src/app/lib/movie/tagging.ts:64-67`（`resolvePlace(opts.country ? \`${place} ${opts.country}\` : place)`）
- 根因见 `src/app/lib/skills/resolvePlace.ts:44`（`limit=1`）

**效果**：实测 Mapbox：「利马」→塞浦路斯利马索尔（错）；「利马 秘鲁」→秘鲁利马 `[-77,-12]`（对）。故事地/取景地「同名异地」钉错洲修复，书架与电影地图落点正确。

### 2.4 心情贴删除键 hover-only → 触屏永远点不到：常显半透明改造

**问题**：心情贴右上角删除 × 用的是 `hidden group-hover:flex`——只在鼠标 hover 时出现。但手机框/触屏**根本没有 hover 事件**，× 永远不显示；缩小态又只是一个 magenta 圆点（点击仅 flyTo 放大、无删除入口），导致整条心情贴在触屏端**完全无法删除**（用户反馈「秘鲁那个紫点删不掉」）。

**为什么难**：它在开发者的鼠标环境下永远正常，只有在目标设备（手机/PWA）上才暴露，属于环境差异引发的功能性死角，桌面开发极易漏掉。

**解决方案**：把删除 × 从 `hidden group-hover:flex` 改为 `flex opacity-50 group-hover:opacity-100 transition-opacity`——常显半透明、鼠标悬停时高亮，触屏一点即删。保留 `onPointerDown stopPropagation` 防误触发 flyTo。一个 className 改动，把「依赖 hover 才可见」的隐式前提换成「始终可见、hover 只做视觉强化」，消除触屏死角。

**代码佐证**：
- commit `25eece8`
- `src/app/components/MyMapTab.tsx:848`（className 由 `hidden group-hover:flex` → `flex ... opacity-50 group-hover:opacity-100 transition-opacity`）

**效果**：触屏/手机框/PWA 上心情贴删除键始终可点，「紫点删不掉」修复；桌面端仍保留 hover 高亮的视觉反馈，不损体验。

### 2.5 持续部署后旧 hash chunk 消失 → 切 tab 白屏：懒加载一次性强刷自愈

**问题**：持续部署模型下，每次发版产生新 hash 的 JS chunk，旧 chunk 从服务器消失。一个挂着没刷新的页面（用户长时间停留），首次切到照片/智能体 tab 时才按需 `import()` 那个已不存在的旧 chunk → `import()` reject。`React.lazy` 对此无内建恢复，外面没有 ErrorBoundary 就直接白屏。

**为什么难**：这是持续部署 + 代码分割 + 长停留会话三者叠加才暴露的故障，本地开发几乎遇不到，demo 现场尤其致命。

**解决方案**：包一层 `lazyRetry` 高阶函数：`import()` 成功就清掉 sessionStorage 标记；catch 到加载失败时，若本会话还没强刷过（sessionStorage 无 `chunk-reload` 标记），就置标记 + `location.reload()` 拉到新的 index.html + 新 hash 清单，并返回一个永不 resolve 的 Promise 让组件挂起等页面重载；若已经强刷过一次仍失败（说明是真故障而非陈旧 chunk），则 throw 交给 ErrorBoundary 显示可见兜底。sessionStorage 标记保证「最多自愈强刷一次」，杜绝重载死循环。三个 tab（PhotosTab/MyMapTab/MusicAgentsTab）全走 `lazyRetry`。

**代码佐证**：
- `src/app/App.tsx:10-22`（lazyRetry：catch 中 sessionStorage `chunk-reload` 一次性 guard + `location.reload()` + 已刷过则 throw 给 ErrorBoundary）

**效果**：持续部署后长停留页面切 tab 不再白屏：陈旧 chunk 触发的 import 失败被自动强刷一次拉到新版本自愈，真故障则降级到 ErrorBoundary 可见兜底，而非白屏死。

### 2.6 瞬时故障被一次性 catch→null 白丢：withRetry 指数退避 + 类型化白名单（4xx 不重试）

**问题**：项目里到处是 fallback（edgeSafe 三级回落、enrichJSON 边→云、舱壁式 catch→null），但缺「同目标退避重试」这条正交轴。云脑 qwen-plus 偶发抖动/429/5xx 这类**瞬时**故障，会被现有的一次性 catch→null 直接当成失败白丢——本该重试一次就成功的请求，被舱壁吞掉返回 null，用户看到的是「功能莫名失灵」。

**为什么难**：难点不在写个 for 循环重试，而在**纪律**——必须用类型化白名单区分「该重试的瞬时故障」和「重试也白搭的真 bug」：4xx 参数错重试只是浪费配额和延迟，把真 bug 反复重试更是雪上加霜。

**解决方案**：借鉴 langchain `RunnableRetry`，把容错拆成正交两轴：`retry`（治瞬时）⊥ `fallback`（治持续/降级），补上缺失的 retry 轴。`withRetry` 做指数退避（`baseMs*2^i`，上限 `maxMs`）+ jitter（`exp/2 + random*exp/2`，错开重试洪峰）；`retryOn` 用类型化白名单 `isTransient` 决定是否重试：`HttpError` 只在 429/>=500 时重试、4xx 不重试；`AbortError`（用户主动取消）不重试；其余网络错/超时视为瞬时重试。接进 `enrichJSON`：关键是 `if(!r.ok) throw new HttpError(r.status)` 让 5xx/429 进重试通道，外层仍保留 catch→null 舱壁。retry 与 timeout 正交——把「带超时的单次调用」整个传进来，每次 attempt 各自超时。

**代码佐证**：
- `src/app/lib/skills/withRetry.ts:23-49`（withRetry 指数退避 + jitter；isTransient：429/5xx 重试、4xx/AbortError 不重试）
- `src/app/lib/skills/enrichEntity.ts:31-41`（enrichJSON 用 withRetry `attempts:3 retryOn:isTransient` + r.ok 检查 throw HttpError + 外层 catch→null 舱壁）

**效果**：云脑 qwen-plus 偶发 429/5xx/网络抖动不再被一次性吞掉，自动退避重试最多 3 次救回；同时 4xx 参数错/用户取消不做无谓重试，既不浪费配额也不掩盖真 bug；舱壁与重试两层并存，重试耗尽才回落 null 走调用方兜底。

### 2.7 坏存档让 recordSignals 在原始值上崩：loadProfile/heartbeat 损坏存档护栏

**问题**：长期画像 `pe.profile.v1` 与 heartbeat state 持久化在 localStorage，这是用户可手改、跨版本会演化、隐私模式会抛异常的不可信输入。若 `domains` 被旧版本/手改/损坏存成了 `7`/`'x'`/数组而非普通对象，后续 `recordSignals` 直接在原始值上赋属性（`profile.domains[domain]=...`）会抛错连锁崩溃；heartbeat 的 cursor 被改成负数/NaN/浮点，`pool[负/NaN]` 取到 undefined → 渲染空建议。

**为什么难**：这些是「加载历史状态」的边界，正常使用永不触发，但一旦命中就是启动即崩的硬故障，且 localStorage 跨版本/跨设备同步会真实制造脏数据。

**解决方案**：在每个持久化模块的 `load()` 里做结构化校验 + 回落：① `profile.load()` 校验 `p` 是对象且 `p.domains` 是普通对象（`typeof==='object' && !==null && !Array.isArray`），任一不满足就回落 `empty()`，从源头杜绝 recordSignals 在原始值上赋属性崩；② `heartbeat.load()` 把 cursor 用 `Math.max(0,Math.floor(Number(s.cursor)||0))` 规整为非负整数、dismissed 用 `Array.isArray` 守为数组；③ 所有 localStorage 读写包 try/catch，隐私模式/容量超限时静默降级到内存仍可用。三处用的是同款「损坏存档护栏」纪律。

**代码佐证**：
- `frost-agent/harness/profile.ts:31-40`（load 校验 `typeof p.domains==='object' && !null && !Array.isArray` → `empty()`）
- `frost-agent/harness/heartbeat.ts:19-27`（cursor `Math.max(0,Math.floor(Number||0))`、dismissed `Array.isArray` 守、try/catch 降级）

**效果**：localStorage 被旧版/手改/跨设备同步污染成脏数据时不再启动即崩：画像回落空白、心跳游标规整、隐私模式降级到内存；recordSignals 不会在原始值上赋属性崩，心跳不会因 `pool[负/NaN]` 渲染空建议。

## 三、端侧性能：把 120s 压到 1~2s

### 3.1 端侧整理相册：单张 120s+ 压到 1~2s 的认知漏斗

**问题**：在纯浏览器（iOS PWA、无原生层、WASM/WebGPU）里批量筛几百张相册，每张直接喂端侧 CLIP/视觉模型，prefill 几 MB 原图，单张要 120s+，几百张直接卡死且爆内存。

**为什么难**：既要快、又不能牺牲判定质量，还要在低算力手机端稳定不 OOM。

**解决方案**：用「认知漏斗」把贵的算力压到最后、最少地用：① 先按 EXIF 金信号（相机字段/GPS/截图软件名/一致性）和像素废片、资料投票这些**便宜信号**确定性地分流，绝大多数照片在这一层就定了类型，根本不进模型；② `screen.ts` 里 `wantVision` 是一个收紧的布尔门：只有「资料临界（`isUtilityProb 0.3~0.7`）/钉候选（有相机 + GPS）/无相机信息」这三类不确定的图才真的跑 CLIP，且 `known?.hadVision` 命中缓存就跳过；③ 发图前在前端 canvas 把原图缩到最长边 ≤512px（`downscaleForVision`）、解码统一缩到 `maxSize=256`，prefill 开销大降；④ 端侧 CLIP 走浏览器内 transformers.js（WASM/WebGPU），原图连本机服务都不发。

**代码佐证**：
- `src/app/lib/photo/screen.ts:38-42`（wantVision 收紧门控 + hadVision 缓存跳过）
- `src/app/lib/photo/features.ts:13-35,73-108`（EXIF 金信号 + utilityVote 多信号投票）
- `src/app/lib/imageDownscale.ts:21-46`
- `src/app/lib/skills/browserVision.ts:82-105`（WebGPU CLIP）

**效果**：单张从 120s+ 压到 1~2s 量级：大多数照片走便宜信号秒判、不进模型；只有少数临界张跑端侧 CLIP，且原图缩到 512px 后 prefill 大降。整本相册批量筛选在手机端可用，且原图不出端、隐私零外泄。

### 3.2 批处理 OOM：canvas/bitmap/原图必须「跑完即弃」

**问题**：手机浏览器堆内存很小，一张原图解码进 canvas 就是几十 MB；几百张串行处理时，只要 canvas/ImageBitmap/objectURL 有一处没及时释放，或被挂到结果对象上跨阶段留存，内存会单调上涨直到页面崩溃。

**为什么难**：要在一条多阶段流水线里保证「任何提前返回路径都释放、任何中间产物都不进主上下文」。

**解决方案**：全链路贯彻「原图/canvas 跑完即释放」：① `decode()` 里用统一的 cleanup 闭包，无论从 `createImageBitmap` 成功路径、`<img>` 兜底路径、还是任一提前 return，都 `close()` bitmap / `revokeObjectURL`，杜绝释放遗漏；② `screen.ts` 的 Handoff Contract——阶段间只传 `PhotoFeatures`/`PhotoResult` 这种小结论对象，canvas 不挂到 result 上，出了 for 循环作用域立即可 GC；③ 持久化只写派生小结论（类型/分值/坐标），原图/canvas/embedding 在落库结构里压根没有字段位；④ 末尾 `delete r.features` 主动释放特征。

**代码佐证**：
- `src/app/lib/skills/browserVision.ts:22-51`（统一 cleanup 闭包，所有路径释放 bmp/objectURL）
- `src/app/lib/photo/screen.ts:42-43,64-68`（canvas 出作用域即 GC、只写小结论、delete features）
- `src/app/lib/photo/features.ts:110-128`

**效果**：几百张串行批处理内存平稳、不再单调上涨爆崩；HEIC/坏图解码失败也走舱壁跳过（不中断整轮），整本相册可一次跑完。

### 3.3 视觉模型实例 + 解析的稳健性：防「跑飞」与脆弱单规则误判

**问题**：端侧小模型（WASM/WebGPU）首次要从 CDN 下权重、可能超时，中途某张推理可能挂起，任一阻塞都会让整条批处理「跑飞」卡死；同时早期用单一像素规则（`colorful<0.18 && contrast>0.55`）判资料图极脆，16:9/4:3 的真实横图被大量误判成截图。

**为什么难**：端侧不可靠环境下要又快又不卡死、又判得准。

**解决方案**：① 每次推理都用 `withTimeout` 包死：CLIP 加载封顶 40s 自动跳过、单张 classify 15s 超时返回 null，绝不无限挂起；失败后 `tried` 复位、不永久禁用，联网恢复可重试；管线（pipe）成功后复用、未就绪一律优雅降级到快速分析，模型失败不挡打分。② 把脆弱单规则换成 `utilityVote` 多信号投票：量化颜色唯一色数 + 最大单色块占比 + 宽高比命中屏幕分辨率 + 无相机 EXIF 等多票加权，并加关键护栏——没有任何「硬屏内特征」时即便宽高比命中也封顶 0.3，不武断判资料。两段式 CLIP 里 STAGE2 瞬时失败也不丢已确认的 `isReal`。

**代码佐证**：
- `src/app/lib/skills/browserVision.ts:15-19,82-114`（withTimeout 40s/15s + tried 复位 + 降级）
- `src/app/lib/photo/features.ts:76-108`（utilityVote 多信号投票 + hardScreenish 封顶护栏）
- `src/app/lib/photo/vision.ts:42-47`（STAGE2 失败不丢 isReal）

**效果**：端侧模型超时/不可达不卡死、不白丢、可重试；资料图误判率显著下降（16:9/4:3 实拍横图不再被误杀），判定既稳又准。

### 3.4 长期画像 fnv fingerprint 缓存：口味未变就不请云脑

**问题**：每个 agent 跑完都往长期画像累加信号，「一句话口味气质」要云脑（qwen-plus）润色生成。若每次进画像页/每轮都请一次云脑，既慢（网络往返）又烧 token 成本；但又不能简单时间缓存——口味真变了得刷新，口味没变就该一直命中。

**为什么难**：要廉价且精确地判断「口味是否实质变化」。

**解决方案**：做一个画像签名作为缓存键：`profileFingerprint()` 只取每个 `domain.field` 的 top6 标签（忽略尾部噪声和计数微抖），排序拼接后过 FNV-1a 哈希得到短签名。`summarizeTaste` 先算 summary、再算 fp，命中已落盘的 `{fp,text}` 就直接返回缓存文本、完全跳过云脑；只有签名变了（口味实质变化）才真的请一次云脑并回写。FNV 是纯整数位运算（`Math.imul` + 异或），零依赖、微秒级。`getCachedTasteLine` 给 memoryRouter 注入时也只读缓存、绝不触发云脑。

**代码佐证**：
- `frost-agent/harness/profile.ts:115-128`（fnv + profileFingerprint top6 签名）
- `profile.ts:144-163`（summarizeTaste 命中 fp 跳过云脑）
- `profile.ts:135-142`（getCachedTasteLine 只读缓存）

**效果**：口味稳定期「一句话气质」零云脑调用、即时返回，省 token 成本与网络延迟；只在口味真变时才请一次，既省又新鲜（成本护盾）。

### 3.5 地图叠层渲染：视口裁剪 + 上限 + 节流重投影，数百点不卡

**问题**：地球放大到街区时，数百张照片拍立得 / 621 首歌落点 / 心情贴都是 DOM 叠层，且每次平移（zoom 不变）都得按新投影重算每个点的屏幕坐标。若每帧把所有点全量 project + 渲染，数百个 DOM 节点 + 每帧重投影会让平移卡顿掉帧。

**为什么难**：大量地理锚定 DOM 元素要跟随地图丝滑平移。

**解决方案**：四招组合：① **视口裁剪**——渲染前先取 `map.getBounds()`，只对 `b.contains([lng,lat])` 在视口内的点 project + 出 DOM，视口外的根本不建节点；② **硬上限**——照片叠层 `out.length>=70/130` 即 break、歌曲落点 `SONG_CARD_MAX=80` 即 break，封死最坏情况节点数；③ **move tick 重投影**——纯平移时 zoom 不变，用 `useReducer` 的 tick 强制重渲染来更新 project 位置，而不是各点自己监听；④ 拖动落点用 `requestAnimationFrame` 节流 setData，松手才 commit 落盘 + 刷新底层源；⑤ 缩小到背面的点用球面中心角 `>78°` 直接 `return null` 不渲染。

**代码佐证**：
- `src/app/components/MyMapTab.tsx:680-736`（getBounds 视口裁剪 + `out.length>=70/130` 上限）
- `MyMapTab.tsx:739-797`（`SONG_CARD_MAX=80` + 视口裁剪）
- `MyMapTab.tsx:459-505`（makeDrag rAF 节流 setData，松手才 commit）
- `MyMapTab.tsx:297-310`（move/zoom → tick 重投影）
- `MyMapTab.tsx:632,801`（`centralAngleDeg>78` 背面剔除）

**效果**：放大到街区数百个候选点只渲染视口内 + 封顶数十个 DOM，平移/拖动按 rAF 节流不掉帧；背面点不渲染，地球缩放平移保持丝滑。

### 3.6 SONG_MARKERS 模块级一次算 + 标记图层 canvas 图标预生成

**问题**：621 首歌散射到城市内坐标（确定性环形散射 + 文本裁剪到句子边界）是有一定成本的纯计算；若放进组件渲染或 `useMemo`，每次 MyMapTab 重渲染（平移每帧 tick 都重渲）都可能重算，纯浪费。同理 mapbox symbol 图层的方块图标若每次都画，也是重复开销。

## 四、长期记忆与隐私边界

### 4.1 回流断裂闭环：产出反哺画像，关掉「越用越不懂你」的单向死循环

**问题**：长期记忆的真正难点不是「读」而是「写回」。常见做法是 agent 跑完只产一次结果（钉个点、给个推荐）就丢，画像永远停在冷启动播种态——heartbeat 还在建议你「整理相册/记一笔」，你照做了，画像却纹丝不动，形成「让你产出→产出不回写→下次还是不懂你」的单向死循环。

**为什么难**：要闭环就得在每个领域 agent 的「确认钉」动作上精确地接一根回流线，且必须解决三个真问题：回流什么（绝不回流隐私）、何时回流（避免重复确认导致计数虚高）、坐标这类非文本信号怎么变成可累加的标签。

**解决方案**：在 `profile.ts` 暴露统一的 `recordSignals(domain, fields)` 累加入口（同标签合并 + 按热度排序 + 每字段截断 50），再让六个领域各自在「首次钉」时调它（movie/book/travel/photo/music 的 pin 都接上）。关键三处取舍：① **选择性回流**——电影/书只回流国别/导演/作者/类型，故意不回流 cast 演员表，「避免画像变社交图谱」；② **幂等护栏**——只在 `r.reason !== 'exists'` 即首次钉时回流，重复确认不再累加；③ **异构信号归一**——照片没有文本标签，用坐标反查出的 `city` 作为信号回流，且「只回流非空，避免写 noise」。`profileSeed` 用 `SEED_VERSION` 幂等播种，`getProfileSeedVersion() >= SEED_VERSION` 直接跳过。

**代码佐证**：
- `src/app/lib/movie/pin.ts:29-37`（首次钉回流 + 故意不回流 cast）
- `src/app/lib/photo/geoPin.ts:67`（坐标反查 city 回流，注释关掉单向死循环）
- `frost-agent/harness/profile.ts:54-73`（recordSignals 累加）
- `src/app/lib/profileSeed.ts:16`（SEED_VERSION 幂等）

**效果**：六个领域 agent 的产出全部反哺同一份画像，实现「越用越懂你」的真闭环；选择性回流让画像保持是口味画像而非社交图谱；幂等播种 + 首次钉守卫保证计数不虚高、重复确认零副作用。

### 4.2 双层记忆各管一摊：会话记忆（短）与跨会话画像（长）正交解耦

**问题**：把「上一句说了什么」和「这个人长期是什么口味」混在一个记忆里，会两头不讨好：塞太多历史轮次会撑爆 context 且让模型被旧对话带偏，只留最近几轮又会跨会话失忆。

**为什么难**：两者的生命周期、存储介质、隐私级别、刷新频率完全不同（会话记忆是易失的、明文的、每轮变；长期画像是持久的、脱敏的、缓变），硬塞一层必然在某个维度上做错。

**解决方案**：刻意拆成两个平行模块：`memory.ts` 只管会话内最近 `MAX_TURNS=6` 轮，纯文本拼 chat_history，不落盘；`profile.ts` 管跨会话沉淀，只存「偏好标签的计数」（导演/作者/艺人/流派/城市…）、不存任何原文。再往上，情绪足迹是第三条独立通道——`mood/retrospect.ts` 注释明确「心情是独立通道，不走 recordSignals」「绝不碰 ProfileDomain」，只读 geoStickers。`memoryRouter.assembleMemory` 在读出口把这几路（叙事 line / 评分偏爱 loved / 情绪 moodTrace / 标签 profile）分通道拼装，各自有无独立判断，谁空谁不注入（省 token）。

**代码佐证**：
- `frost-agent/harness/memory.ts:1-3`（「memory.ts 管会话内最近几轮、profile.ts 管跨会话口味」）+ `MAX_TURNS=6`
- `frost-agent/harness/profile.ts:1-4`（结构化脱敏画像、只存标签计数）
- `src/app/lib/mood/retrospect.ts:3`（情绪独立通道不走 recordSignals）
- `src/app/lib/memoryRouter.ts:34-44`（四通道分别判空拼装）

**效果**：短期上下文不撑爆 context、不被旧对话带偏；长期画像持久脱敏、缓变可缓存；情绪通道独立不污染口味画像；读出口一行 `assembleMemory()` 统一拼装，新增 agent 走这里就不会漏接记忆。

### 4.3 原图/画像不出端：端云分脑的硬隐私边界

**问题**：端侧 PWA + 云脑（通义 Qwen）的架构里，最敏感的两类数据是用户原图（可能含证件/卡号/人脸）和长期画像（刻画一个人的口味档案）。隐私边界一旦只靠「提示词让模型别外传」就形同虚设——必须在代码结构上保证原图根本没有路径流向云、画像根本没有路径流向端侧小模型。

**为什么难**：这种「不接触」是负向约束，容易在某个 agent 新加读图/注入逻辑时被悄悄破坏（曾经脱敏正则在 movie/travel/visionExtract 各有一份、各自演化）。

**解决方案**：两道结构性铁律。① **原图只进端侧、绝不出端**：`visionRead` 是全项目唯一读图收口，原图只在这里碰一次端侧 `edgeSafe.vision`（浏览器内 / 服务端的 Qwen 视觉模型，对应 ollama qwen2.5vl:3b / MNN Qwen2.5-VL-3B），「一个字节都不写持久层、不上云」，读完即弃；端侧未就绪/读不出就返回空串走手填兜底，绝不把原图送云；脱敏正则 `REDACT`（证件 18 位→卡号→手机，顺序避免互吃字）只在这一处维护一份，与端侧提示词「别输出敏感号」互为双保险。② **画像只在云脑侧读**：`profile.ts` 头部写明「本模块只在云脑侧（httpBrain/对话注入）读取，端侧 Selector（/api/edge）一律不接触，画像不出端到端侧模型」，`profileSeed` 同样标注。端侧文本三件套走浏览器内 WebLLM **Qwen3-0.6B**（真端侧、动态 import、不进主 bundle），与云脑画像物理隔离。

**代码佐证**：
- `src/app/lib/skills/visionRead.ts:5-12`（唯一收口、用完即弃、不落持久层）+ `18-23`（REDACT 三档脱敏）+ `47`（读不出返回空串走手填）
- `frost-agent/harness/profile.ts:6-7`（隐私边界注释：端侧 Selector 不接触）
- `src/app/lib/profileSeed.ts:5`（画像只供云脑侧）
- `frost-agent/edge/webllmEdge.ts:1-4`（端侧 Qwen3-0.6B 真端侧）

**效果**：原图最多在端侧模型前停留一次、零持久化零上云，且单一收口 + 单份脱敏正则杜绝各 agent 各演化一份的漂移；画像与端侧小模型物理隔离，即便端侧被攻破也拿不到口味档案；构成「原图不出端 + 画像不出端到端侧」的双向隐私边界。

### 4.4 「读得多≠真正爱」：按评分加权让真口味浮出、对抗数量偏置

**问题**：纯按数量统计的画像有系统性偏差：读得多/看得多的大国（中国/美国/日本）会天然盖过读得少但打分极高的小众领域（如拉美魔幻现实主义）。如果就这么注入云脑，推荐和叙事会一直被高频低分内容主导，把用户「真正钟爱但量少」的口味埋掉。

**为什么难**：这是个隐蔽的真问题，画像看起来「数据很多」却恰好把最该突出的口味稀释了。

**解决方案**：两层对冲。① **入库即加权**：movie/book pin 回流时按真实星级复制信号——5★×3、4★×2、其余×1（`rep(arr)=arr.flatMap` 复制 w 份），让高分作品在 `recordSignals` 计数里权重更高。② **读出口另算一份评分视图**：`taste.ts` 不走画像的数量统计，直接从真实书目（含国家 + 评分）按「国家→地区」汇总，要求地区有一定量（≥8 本、≥5 本有评分）后按平均评分排序，产出「最偏爱（按地区平均评分，反映真正钟爱而非读得多少）」一行，与「读得最多」并列注入云脑。`memoryRouter` 把这份 loved 作为独立 L2 通道拼进记忆块。

**代码佐证**：
- `src/app/lib/movie/pin.ts:31-32`（`w=5★?3:4★?2:1` + rep 复制）
- `src/app/lib/book/pin.ts:24-25`（同款星级加权）
- `src/app/lib/taste.ts:38-50`（按地区平均评分排序的「最偏爱」、阈值 `n≥8/rated≥5`）
- `src/app/lib/memoryRouter.ts:36+40`（loved 作 L2 通道注入）

**效果**：拉美魔幻现实主义这类量少分高的真爱能从大国高频内容里浮出来；星级加权 + 评分视图双管齐下，云脑拿到的画像同时反映「读得多」和「真正爱」两个正交事实，推荐/叙事不再被数量偏置主导。

### 4.5 记忆即空气：成本护盾（fingerprint 缓存）+ 注入规则，让长期记忆便宜且不出戏

**问题**：把长期记忆注入每次对话有两个隐性代价：① 每次都请云脑把画像润色成「一句话口味气质」会持续烧 token、加延迟；② 模型若把记忆当对话内容复述（「我记得你之前说过…」）会让用户出戏、甚至用旧记忆去纠正反驳用户当前这句话。

**为什么难**：这两点不解决，长期记忆就是个又贵又惹人烦的负担。

**解决方案**：① **fingerprint 成本护盾**：`profileFingerprint()` 只取每字段 top 标签做 FNV 签名，签名不变=口味没实质变化，`summarizeTaste` 命中缓存（`NARR_KEY` 里 fp 相同）直接返回，跳过云脑；memoryRouter 注入时用 `getCachedTasteLine()` 只读缓存、绝不触发云脑，刷新由后台 fire-and-forget 的 `ensureNarrative` 做（`narrPending` 去重，不阻塞当前回答）。② **「记忆即空气」注入规则**：`MEMORY_AIR_RULES` 明确要求「自然融进回答、不要当对话内容复述、不要说我记得/你之前说过、若与用户当前这句冲突一律以当前对话为准、绝不用旧记忆纠正反驳用户」；无任何记忆时连规则都不注入，省 token。

**代码佐证**：
- `frost-agent/harness/profile.ts:122-128`（profileFingerprint FNV 签名）+ `148-163`（fp 命中缓存跳过云脑）
- `src/app/lib/memoryRouter.ts:15-18`（MEMORY_AIR_RULES 不复述/冲突以当前为准）+ `24-28`（ensureNarrative 后台保鲜不阻塞）+ `43`（无记忆返回空串）

**效果**：口味叙事只在画像实质变化时才请一次云脑、平时零成本读缓存，注入不阻塞当前回答；模型用记忆但不出戏、不谄媚、冲突永远以当前对话为准，长期记忆从「又贵又烦的负担」变成「便宜且自然的空气」。

## 五、编排泛化与可观测

### 5.1 注册表化内核 dispatch：新增意图/动作不改 router 与 validator

**问题**：多智能体 harness 最容易腐化的地方是总控：经典写法把意图分发写成 `switch(intent){case 'tour':...}`，动作校验写成一串 `if(type==='play')...`。每加一类对象（电影/书/行程/心情/音乐）就要回头改两处内核代码、碰整个团队的热区文件。

**为什么难**：既是合并冲突重灾区，也违背最小权限——校验白名单散落在 if 链里很难审计。要做到「新增一类 skill 内核零改动」，必须把控制流反转成数据（注册表），同时保住两条边界：未注册意图安全回退、未注册动作一律拒绝。

**解决方案**：`router.dispatch` 不再 switch，改成 `getIntentHandler(intent)` 查注册表：命中就调，未命中回退 `runGeneral`。意图处理器靠 `registerIntent(intent,handler)` 声明式登记，各 skill 的 `switch/tour/open_dj` 等全是注册进去的，不是内核硬编码。动作校验同构：`validateActions` 逐个动作查 `validators[type]`，未注册类型直接 rejected「未知动作类型」——默认不放行的最小权限闸；新增校验器走 `registerActionValidator(type,fn)`，`switch_city` 还在校验器里核对城市真在资源库 `RADIO_CITIES` 才放行。两张表把内核 dispatch 从「认识每一种 case」降级成「只会查表 + 守边界」。

**代码佐证**：
- `frost-agent/harness/intentRegistry.ts:19,34-38`（registerIntent/getIntentHandler + router 查表回退）
- `validator.ts:15-17,40-50`（registerActionValidator + 未注册即拒）
- `router.ts:34-38`（dispatch 无 switch）

**效果**：新增一类 curator/skill = 注册一个 IntentHandler + 若干 ActionValidator，`router.ts`/`validator.ts` 内核一行不动，避开 `server.mjs`/`MyMapTab` 这类多窗口协作热区；动作侧默认拒绝未知类型，子 agent 只能「建议」、越界动作进不了地球，边界可审计。

### 5.2 matchCity 确定性护栏：云脑把国家/地域脑补成城市时强制降级

**问题**：把意图判断交给云脑（qwen-plus）读自然语言是泛化的关键，但 LLM 有个稳定的失败模式：抽城市时会自作主张。「放点日本的音乐」本该是按国家点歌（`open_dj`），云脑常脑补成「切到东京」（`switch_city`）；「放点欧洲的」会被切到某座欧洲城。一旦切城动作落地，用户听到的是完全错的城市。

**为什么难**：纯靠 prompt 反复强调「国家不是城市」压不死——这是概率模型，总有抖动漏过去。可观测/泛化层最难的不是让模型聪明，而是给聪明的模型套一道不会抖动的确定性兜底。

**解决方案**：在 LLM 路由出口加一道程序化复核：只要云脑判了 `intent==='switch'`，就回头用确定性的 `matchCity(text)` 核对——这座城必须以字面量真出现在用户原话里（`matchCity` 遍历 `RADIO_CITIES` 做 norm 后的 `includes` 子串匹配）。没命中（多半是国家被脑补成城）→ 直接把 intent 降级成 `open_dj` 点歌、清空 city、并在 reason 里写明「原话未点名具体城市，按点歌处理」；命中了 → 以原话那座城为准，盖掉云脑可能抽歪的 city。护栏是字符串包含判定，零随机性，把 LLM 的概率误判收敛成确定结果。

**代码佐证**：
- `frost-agent/harness/llmRoute.ts:44-51`（switch 出口 matchCity 复核→降级 open_dj/盖 city）
- `agents/switch-handler/index.ts`（matchCity：norm + includes 子串匹配 RADIO_CITIES）

**效果**：「放日本的音乐」稳定走出日本歌单而非误切东京；治住「国家被脑补成城市」这类 LLM 系统性抽歪；泛化（云脑读懂长尾问法）与可靠（护栏不抖）兼得——同一思路也复用到 `capture/route.ts` 的判域护栏。

### 5.3 FrostBus 统一可观测事件树：一套带 runId/parentId 的事件喂多个消费者

**问题**：端云双脑多 agent 跑起来，一次请求会串 `router→agent→skill→enrich` 好几跳，原来只有零散 trace + health，没有统一事件词汇，无法把一次运行串成可追踪的树，更没法既喂调试面板又驱动前端展示。

**为什么难**：直接搬 langchain 的 `CallbackManager`（2826 行）对单浏览器单 React 树是杀鸡用牛刀；但若每个 UI 各自埋点，事件口径会发散、互相打架。难点是：用极小的地基，既统一事件词汇、又让一份事件同时被多个互不知情的消费者订阅，且观测层任何抛错都绝不能打断业务主流程（舱壁）。

**解决方案**：剥到约 40 行的事件总线：统一 `FrostEvent` 词汇带 `runId/parentId` 串成调用树，`frostBus.emit` 先入一个 1000 容量的 ring buffer 再通知所有 handler，且每个 handler 用 try/catch 包住——观测层抛错只吞掉、不波及业务（舱壁）。ring buffer 专门解决「订阅前已发的 start/首阶段事件」被晚挂载的订阅者补上：`frostBus.recent(runId)` 先 seed 历史再增量订阅。同一份事件被两个互不知情的消费者吃：`RunTrace` 按 `runId/parentId` 过滤、画单次运行的内联思考痕迹；`RunDrawer` 只收 agent 根事件、维护最近 40 次运行的全局轨迹抽屉。`startAgentRun(label)` 封装 start/phase/end 三件套，7 个运行页统一接入免重复样板。

**代码佐证**：
- `src/app/lib/observe/bus.ts:15-26,36-39`（事件树 + 舱壁吞错），`41,57-66`（recent seed + startAgentRun）
- `RunTrace.tsx:23-31` 与 `RunDrawer.tsx:15-19`（同一事件流两个独立消费者：内联 vs 全局）

**效果**：一次 `router→agent→skill` 跑动在 RunTrace 显成可读的分步痕迹（命中/降级/重试徽章）、同时在 RunDrawer 沉成全局运行历史；新增 agent 只调 `startAgentRun` 即自动接入观测，无需各自埋点；观测层崩了业务照跑。

### 5.4 真 SSE 逐 token 流式：additive 路由 + per-token 不发 FrostBus 的取舍

**问题**：原来「先整段生成再打字机模拟」是假流式——用户要等整段算完才开始看字。要做真·云脑逐 token 吐字，需打通三段：DashScope/Qwen OpenAI 兼容上游 `stream:true` 的 SSE → Node 服务端透传 → 浏览器 fetch reader 逐块解析。

**为什么难**：两端都踩同一个坑——网络分片不按 SSE 行边界切，一个 `data:` 帧可能被劈成两块。还有个反直觉的设计张力：刚建好的 FrostBus 是「所有过程都该上事件树」，但 per-token 若每个都 emit 一条会瞬间灌爆 1000 容量的 ring buffer——可观测的「全都记」原则在 token 粒度上必须破例。

**解决方案**：新增独立 `/api/frost-llm-stream` 路由，additive 不改原非流式路由：上游 `stream:true`，逐 token 解析 `choices[0].delta.content` 透传 `data:{token}`，`[DONE]→data:{done:true}`，非 2xx/无 key 走 `data:{done:true,error}` 让前端兜底。两端都用同一套「半行缓冲」抗分片：`buf.split('\n')` 后 `lines.pop()` 把可能不完整的末行留到下一块（server.mjs 与 streamComplete.ts 对称实现）。客户端 `streamComplete` 逐 token 累计 full 并回调 `onToken`，AgentChat 消费它做真打字。关键取舍写进注释并执行：per-token 只走 `onToken` 回调、不发 FrostBus，避免每 token 一条灌爆 ring buffer；失败抛错由调用方兜底回非流式 `complete`（舱壁）。

**代码佐证**：
- `server.mjs:91-130`（additive SSE 路由 + delta.content 透传 + 半行缓冲 114-115）
- `src/app/lib/streamComplete.ts:28-29,35-36`（对称半行缓冲 + onToken），`2-3`（per-token 不上 FrostBus 的明示取舍）
- `AgentChat.tsx`（消费方）

**效果**：AgentChat 真·逐 token 显示云脑吐字，首字延迟从「整段算完」降到「首 token 到达」；新路由不动旧非流式路径零回归；在 token 粒度上对可观测原则做了有意识的破例，既保住流畅又不撑爆事件缓冲。

### 5.5 skillForge 学习型技能的白名单安全闸：reviewSkill 默认不放行、拒疑似代码

**问题**：让 frost「学」新技能（一句话→云脑拟清单→安装）很有想象空间，但这是自进化系统最危险的入口：若技能能携带可执行代码，等于把云脑生成的内容直接喂进运行时，一句话注入就可能 eval/fetch/外链。

**为什么难**：真·让 agent 自写可执行代码属高风险。难点是在「可学习」和「绝对安全」之间划一条程序能守住的线：技能必须只是声明，不能是代码；而云脑产出的 JSON 你无法假设它干净。

**解决方案**：把技能本质钉死成「触发词→路由到一个已有 agent」的声明式快捷方式，不含也不执行任何代码。`reviewSkill(m)` 是默认不放行的多重闸：`target` 必须 ∈ `ALLOWED_TARGETS` 白名单（只能指向已存在的可运行 agent）；字段走白名单，多一个未知字段就拒；最关键一道——把整个清单 `JSON.stringify` 后做安全扫描，命中 `<script` / `eval` / `require` / `import` / `process.` / `child_process` / `fetch(` / `http(s)://` / 反引号 / `${` 任一即拒「检测到疑似代码/外链」。`proposeSkill` 让云脑只产 JSON，`installSkill` 落地前再过一次同一道闸，双重校验。

**代码佐证**：
- `frost-agent/harness/skillForge.ts:18-27`（ALLOWED_TARGETS 白名单），`32-46`（reviewSkill：target 白名单 + 字段白名单 + 疑似代码正则扫描），`86-88`（installSkill 落地前再审一次）

**效果**：学习型技能只能是「关键词→已有 agent」的声明式路由，云脑生成内容里任何代码/外链/未知字段在安装前被拦死；自进化的想象力保留，但运行时不接受一行模型生成的可执行代码，把高风险面关在门外。

### 5.6 JOT 统一记一笔判域泛化：云脑判域 + 三层确定性护栏纠偏

**问题**：「一个框记一切」要求云脑把任意一句随手记判进电影/书/行程/心情四域，再调对应现成管线。但判域有几类稳定误判会把记忆钉错：①带截图但文字没作品名时，「帮我记录下这部电影」被判成心情；②「去了京都」这类亲身出行被判成 mood；③「看了波拉尼奥想去圣地亚哥」被作家名带偏判成读书，且把整句喂地理编码器会把智利圣地亚哥拽到哥伦比亚同名小镇（钉错洲）。

**为什么难**：纯靠 prompt 判域接不住这些长尾，且地名抽取的噪声会污染下游定位。

**解决方案**：云脑判域（`classify` 输出 `{domain,place}`，失败一律回落最稳的 mood）打底，再叠三层确定性护栏纠偏：① **领域词护栏**——原话出现「这部电影/影片/剧集」词组强制 movie、「这本书/这部小说」强制 book，且收紧成词组（不含裸「电影/书」）避免「电影博物馆」「软件作者」误伤；② **出行护栏**——`domain==='mood'` 且命中出行动词（去了/去过/出差/旅行…）又能 `geocodeCity(t)` 命中可识别城市，强制 travel；③ **愿望护栏**——「想去/打算去 + 某地」且后面不接看吃玩等动作，不限源域强制 mood 记成「想去」便签。地名抽取上，让云脑从整句只抽规范地名 `place`（还借上下文消歧「波拉尼奥是智利作家」→智利圣地亚哥）喂 `resolvePlace`，而非把整句噪声喂地理编码器。

**代码佐证**：
- `src/app/lib/capture/route.ts:42-66`（classify 云脑判域 + 回落 mood），`80-86`（三层护栏：领域词组/出行动词 + geocodeCity/想去），`99-102,113-118`（只抽规范地名 place 喂 resolvePlace 防钉错洲）

**效果**：四类随手记各自判域钉对（奥本海默→电影取景地、去了京都→行程、有点累→心情）；治住「这部电影 + 截图被判心情」「去了京都被判 mood」「想去被作家名带偏」三类系统误判；地名只抽不灌，智利圣地亚哥不再被拽到哥伦比亚同名小镇。

---

## 结语

把这些难点摊开看，会发现它们指向同一个判断：**Pocket Earth 真正啃下的硬骨头，不在「让模型更聪明」，而在编排与工程**——在 LLM 永远会缺位、会抖动、会脑补、会越界的前提下，用契约面、确定性护栏、纵深超时、注册表、损坏存档护栏、认知漏斗、回流闭环和物理隔离的隐私边界，把一个个概率性的「可能崩」收敛成确定性的「降一级继续」。模型（云 qwen-plus、端 Qwen3-0.6B / Qwen2.5-VL-3B）只是其中一个会失灵的零件；让整套系统在它失灵时依然稳、依然快、依然守得住隐私，才是这份工作的全部分量。
