# 端侧整理照片 agent · 机制设计（多 agent 调研+评审 合成）

## 总览
机制改成「类型优先 + 钉地球正交闸」：每张照片先解码缩图(沿用 decode，长边256)，按性价比依次过四道便宜信号——EXIF金信号(扩 readCaptureDate→readExif 读 Make/Model/曝光/GPS)、像素废片闸(沿用 sharpness<0.08/mean<28)、资料(isUtility)多信号投票、必要时才对少量不确定项跑 CLIP 兜底——得到一张照片的三件事:photoType(类型)、verdict(留/待定/可清理 建议)、pinnable(能否钉地球)。其中 verdict 与 pinnable 是两条正交输出:质量分被彻底降级为只管「废片过滤+簇内选优」,绝不再单独决定留/钉。钉地球是一条独立硬逻辑:实拍类型 ∧ 有真实拍摄坐标 ∧ 簇内代表 才钉;资料型一律不钉、且只标不删。关键纠偏(对两份评审的核心让步):因为 HEIC 在 PWA/Safari 上 GPS 解析失败率高、国内社交平台普遍剥 EXIF,「有 GPS」不能当会让大多数中国/iPhone 用户地球全空的死硬闸——所以坐标降级为「有真坐标→直接钉;读不出但其它信号判定为实拍→进『疑似有地点·待你确认』中间态,引导用户点一下补地点」,而不是冷冰冰判死。所有 PWA 端的判别都是「带置信度的建议 + 显眼纠错入口 + 记住纠正」,默认偏置压倒性地是「宁可放过、绝不误杀珍视照片/纪念截图」。原图与中间画布永不出端,进知识库的只有 {dHash, type, 分数, verdict, pinnable, hasGPS, 经纬度, exif摘要}。

## Q1 筛选标准
最终筛选标准 = 「类型 × 质量 × 价值 × 地理」四维组合,而不是单一质量分(用户实测翻车的根因就是只用质量分):①类型先行——先判这是实拍还是资料/废片,类型决定它走哪套价值标准、默认怎么处置、能否钉地球;②质量退化为废片闸——sharpness/曝光/对比只用来过滤模糊/欠曝/纯色误拍 + 在相似簇里选最优一张,绝不再单独把一张资料截图判「留」;③价值按类型分别算(见 value_by_type)——风景算美学+地理稀有度,人物算有人/主体清晰,资料算可检索性而非美;④地理只作用于「钉地球」这条正交线,不影响留/待定/可清理。判定顺序固定为:解码→EXIF金信号→废片闸→资料(isUtility)投票→(不确定才)CLIP→查重选优→按类型算价值→出 verdict+pinnable。这条顺序把贵操作压到最后、最少,也保证「先分清是什么,再谈值不值/钉不钉」。

## Q2 类型体系（+ 能否钉地球 + 端侧怎么识别）
- **实拍·有地点价值(风景/自然·城市街景/建筑·地标)** — 例：海边、雪山、街道、夜景、寺庙、招牌、菜市场；钉地球：✅
  - 识别：EXIF 有 Make+Model+曝光三元组→强实拍先验;CLIP 子类 landscape/cityscape 高;有真 GPS→直接进钉候选,无 GPS 但判实拍→进『待确认地点』
- **实拍·人与生活(人物/自拍合影·美食·宠物动物·活动现场·日常物品)** — 例：家人合影、一盘菜、猫、演唱会、桌上的花；钉地球：✅
  - 识别：EXIF 相机字段 + CLIP person/food/animal/activity 子类;PWA 无人脸,人物退化为 CLIP person 置信度+主体清晰度近似,原生换 Vision 人脸
- **资料·屏内生成(截图·聊天记录·网图转存·表情包·海报/插画/卡通)** — 例：技术文档截图、聊天截图、漫画海报、网络存图；钉地球：❌
  - 识别：宽高比精确命中屏幕分辨率表 + 大面积单一纯色块/状态栏规律亮块 + 量化唯一色数少(直方图尖峰) + 无相机EXIF + Software 含截图工具名;CLIP screenshot 兜底
- **资料·拍摄的文字(翻拍文档/白板/PPT·菜单·票据收据·证件·名片·二维码·手写)** — 例：翻拍的文件、餐厅菜单、车票、身份证；钉地球：❌
  - 识别：二值化后小连通域文本密度高 + 水平行结构;可能有相机EXIF甚至GPS,但类型闸一票否决钉地球。注:店招/菜单这类有地点意义的,靠纠错入口一键拉回
- **实拍·无地点(待补位)** — 例：判为实拍但 EXIF 被微信/IG/截屏二次保存剥了 GPS、或 HEIC 读不出坐标、或拍摄时关了定位；钉地球：❌
  - 识别：isUtility=false 但 hasGPS=false;不判死,进『疑似有地点·待你确认地点』中间态,引导用户补地点后转正进钉候选
- **废片(重模糊·严重欠曝/过曝·纯色误拍·近重复次优张)** — 例：手抖糊片、口袋误拍、连拍冗余；钉地球：❌
  - 识别：sharpness<0.08→模糊;mean<28→太暗(夜景/星空/虚焦艺术照走低光豁免,见 ziliao_answer 同款豁免逻辑);相似簇内非代表张
- **不确定(低置信,启发式与 CLIP 都没把握 0.35~0.65)** — 例：带风景的截图、拍的有字的招牌；钉地球：❌
  - 识别：投票落 0.35~0.65 一律退『待定』不武断判死,优先跑 CLIP 兜底;仍不确定则交用户,默认偏保留

## per-photo 流水线
1. 闸门0·解码:沿用现有 decode() 缩图到长边256(HEIC 走 createImageBitmap→<img> 兜底),取像素数组+宽高+文件大小。dHash 在这步算出,作为后续稳定幂等键
2. 闸门1·EXIF金信号(最先最便宜最确定):把 readCaptureDate(photoScreen.ts:43)扩成 readExif,除日期外读 Make/Model/Software/FNumber/ExposureTime/ISO/FocalLength/GPSLatitude/GPSLongitude,返回 {capDate, hasCameraFields, hasGPS, lat, lng, softwareIsScreenshot}。做一致性校验:GPS 在海里/时间在出生前/Make 是已知截图工具→标 suspectExif,不盲信
3. 闸门2·废片闸(沿用现状阈值,但只服务清理分支、不参与钉地球价值):sharpness<0.08→模糊废片;mean<28→太暗废片——但先过『低光/艺术豁免』:若 colorful 不低或有少量高光点或 CLIP beautiful 概率不低,夜景/星空/虚焦/雪雾极简不判废片
4. 闸门3·资料(isUtility)多信号投票(便宜,能定大多数资料型):宽高比命中屏幕分辨率 + 最大单一纯色块占比 + 量化唯一色数 + 二值化文本密度+行结构 + EXIF 无相机字段/含截图Software,融合成 isUtility 概率。替换 photoScreen.ts:270 那条脆弱的 colorful<0.18&&contrast>0.55 单规则
5. 闸门4·仅对不确定样本跑 CLIP(贵,靠后,限量;沿用 useModel/modelTopN 限流):对 isUtility 投票模糊(0.35~0.65)或将进钉候选的图,跑改造后 ensemble CLIP——先二分实拍↔资料(纠偏 isUtility),实拍再判子类。CLIP 不可用(离线/镜像挂)时 isUtility 闸降级为『不否决·宁放过』,绝不让拍脑袋启发式单独误杀
6. 闸门5·查重分两档+资料放宽:先按 EXIF DateTimeOriginal 以~10分钟断点时间分桶(无 DateTimeOriginal 的截图/网图单独归『无日期桶』不与有日期照片混簇,避免跨年误删);桶内 dHash 海明≤2→严格重复(留分高一张),3~8→相似簇(只聚类、簇内选最佳、其余标可清理不自动删)。对判为资料型的图放宽/跳过 dHash 去重,防误杀连续聊天截图/同模板表情包
7. 闸门6·时空聚类把散点变经历(新模块 photoCluster.ts):对有 GPS 的实拍候选,时间10分钟分桶→桶内 GPS 空间 DBSCAN(eps 数百米),每簇=一次经历;增量 agglomerative,新照只与时间相邻簇比对并入或开新簇,不重算全库
8. 闸门7·按类型算价值+簇内选代表+反查地名:实拍按 value_by_type 算价值分,资料不进美学排序;每簇选价值最高一张为代表;反查地名先用 EXIF GPS,城市级字典(geoStickers.ts PLACE_COORDS)只做就近显示且明确标『约』,默认走真坐标钉点、地名仅作标签(不靠字典定位)
9. 闸门8·出三件套+只写派生数据:verdict(留/待定/可清理 建议)+ photoType + pinnable;以 dHash 为主键写 IndexedDB {dHash,type,score,verdict,pinnable,hasGPS,lat,lng,exif摘要,clusterId,userOverride},原图/canvas/CLIP embedding 一律不入库

## 决策树（verdict ⟂ pinnable）
verdict(留/待定/可清理)是「建议」,由类型+价值+重复共同决定,且压倒性偏置「宁可放过」:●废片(模糊/太暗/纯色误拍)→『可清理』;●相似簇内非代表的严格重复张→『可清理』(代表张『留』);●实拍类型→价值分≥60『留』、40~60『待定』、<40 且确属像素废片才『可清理』;●资料·拍摄文字(票据/证件/文档/菜单/二维码/手写)→默认『待定·归档』,绝不默认可清理(可能是报销凭证/证件);●资料·屏内生成(截图/网图/表情包/海报)→默认『待定』(注意:不再像初版那样默认『建议清理』——这是对评审的关键让步:纪念截图/和逝者的聊天记录被默认标可清理是比『资料被误判留』严重得多的反向伤害);●仅当一张截图同时满足『dHash 与库内已有严格重复』或『纯临时网图特征极强且无人脸无人名』才给弱『可清理』建议。pinnable 是与 verdict 正交的独立布尔=实拍类型(非资料/非废片/非不确定) ∧ 有真坐标(或用户已补) ∧ 该类型价值达钉地球阈值 ∧ 簇内代表;资料型即使带坐标、即使被判『留』,pinnable 恒 false(类型一票否决优先级高于坐标)。低置信(0.35~0.65)一律退『待定』。所有『可清理』永远只是建议+标记,绝不替用户删原图。」

## Q5 钉地球准则
钉地球 = 类型闸 × 坐标(降级后) × 价值/代表闸,三者组合,但坐标不再是会让地球全空的死硬闸。准则照搬 Apple Places 的『地图契约』——地球上一个点意味着『人在此地、亲手拍下此景』。①类型闸(一票否决):必须实拍组(风景/街景/人物/美食/宠物/活动/日常);资料·屏内、资料·拍摄文字、翻拍实体资料、废片、不确定 一律挡掉——截图的坐标是『屏幕内容的位置』不是『身体所在』,翻拍文档的 GPS 与照片语义无关,类型优先级高于坐标。②坐标(降级,这是对两份评审 HEIC-GPS 痛点的核心修正):有真 EXIF GPS→直接钉,坐标打 source=exif;读不出 GPS 但判为实拍→不判死,进『疑似有地点·待你确认地点』中间态,用户点一下从字典/地图补坐标后转正(source=user),或时空借位(同次行程相邻有坐标的照片借位,source=borrowed,默认不外泄);明确告诉用户『是相册本身缺坐标,不是筛选 bug』。③价值/代表闸:同一次经历只钉簇内最佳一张+簇内张数,不钉几十个重叠点。【簇代表刷新规则(补评审 gap):钉点稳定的是 clusterId 而非具体某张;新批次进来若刷新了某簇最佳,原地更新该簇代表的缩略图引用、坐标基本不变,做到『簇只增、簇代表可被更好的替换』,不与只增不重算冲突】。无 GPS 怎么办:绝不强行钉到错位置;给体面的『待确认地点』状态 + 一键补地点入口 + 时空借位建议,把『地球长期空着』的风险从坐标硬闸手里夺回来。落地:接 mapMarkers.ts 已有的 kind:'photo' marker(含 lat/lng/thumb/full),新增 addPhotoPin 只接 pinnable===true 的簇代表,与 geoStickers 的 mood 贴分层。隐私补丁:分享/截图地球时对 source=exif 的精确坐标做抽稀(降到街区/城市级)+ 家庭/公司高频点识别后默认不外泄——这是比『原图不出端』更被低估的洞。

## Q3 资料型留不留（结论）
结论(直接回答用户最纠结的那一问):资料型一律『不钉地球、不自动删、默认待定·进独立可检索资料分区』,把删的决定权完全交还用户——绝不替用户判它没用,更绝不默认建议删。细化:①一票不钉:类型闸否决,与有没有坐标无关;②默认『待定·归档』而非『可清理』(对评审的关键修正——初版把『屏内生成默认建议清理』当通则会主动建议清掉用户珍视的纪念截图/和逝者的聊天记录,这是会击穿信任的反向伤害,予以取消);③价值不按美算,按可检索/有用性另立一档(票据含金额日期、证件、二维码、文档),原生阶段对这档跑 OCR 抽文本作『可检索价值』另存;④建『资料』分区:可检索、可一键转正、可彻底删,避免隐藏式删除;⑤情感截图豁免:含人脸/含人名/被用户收藏过/反复查看过的截图,从任何『可清理』建议里豁免,默认『待定·留』;⑥『拍的菜单/招牌/票根/站牌』这类有地点意义却文本密度高的——机制承认 PWA 启发式+CLIP 分不开它和『纯翻拍文档』,所以不靠机制硬判,而是给显眼的『一键拉回实拍/钉地球池』纠错入口,且纠错被记住(写 userOverride,回写个性化,见 long_running),用 GPS 是否存在做加权参考而非决定。一句话:资料型的处置原则是『移出地图主线、不删、归档可查、把判断权留给用户、纠错被记住』。

## Q4 各类型的价值标准
- **实拍·风景/街景/地标**：价值=美学(PWA: CLIP『beautiful photo』概率+构图/色彩/清晰 三项加权,初值各 0.3/0.3/0.4;原生: overallScore) + 地理稀有度(该时空簇照片越少越珍贵) + 是否簇代表。地理稀有度的分母明确=『历史进过库的全部照片(IndexedDB 累积)』而非本批,本批量太小时稀有度权重自动调低避免荒谬排序。不靠人脸
- **实拍·人物/自拍合影**：价值=主体存在与清晰(PWA: CLIP person 置信度+主体区域清晰度近似,无法判睁眼/谁是主角——诚实承认 PWA 这档偏弱,默认偏『留』不偏『清理』,原生换 Vision 人脸 landmark/睁眼/正脸/主体占比) + 美学 + 有人=高权重(即便构图普通也倾向留)
- **实拍·美食/宠物/日常物品**：价值=主体清晰 + 色彩饱满 + 适度美学,权重偏主体非全局构图。PWA 无 saliency,用『中心区域清晰度 vs 全局清晰度比值』近似主体清晰,原生换 Vision saliency。簇内挑主体最清晰那张为代表
- **实拍·活动现场**：价值=有人/有主体 + 清晰 + 时近性(稀有事件加权)
- **资料·拍摄文字(文档/票据/证件/菜单/二维码/手写)**：价值≠美学(套美学会得出荒谬排序),而是可检索性/唯一性/近期性:是否票据(含金额日期)/证件等结构化信息、库内是否已有、是否近期拍。PWA 先只做归档+去重(对资料放宽 dHash 防误杀),原生用 OCR 量化。默认归档不删
- **资料·屏内生成(截图/网图/表情包/海报)**：价值=是否唯一/是否近期/是否含人脸或人名(情感豁免)。默认『待定』不进美学排序;仅库内严格重复或纯临时网图特征极强且无情感信号才给弱清理建议——绝不默认建议清理珍视截图
- **废片**：无价值维度,纯由像素废片闸(模糊/太暗/纯色误拍/相似簇非代表)判定,统一『可清理』建议(仍只标不删);夜景/星空/虚焦/雪雾极简走低光豁免不进此档

## Q6 长期本地任务

## PWA 现在 vs 原生以后
同一套『type + 价值 + pinnable』schema 契约不变,只换底层实现。【PWA 现在能做到】EXIF金信号(exifr 扩读 Make/Model/曝光/GPS,零模型成本,当前最该补)、宽高比命中屏幕分辨率、纯色占比/唯一色数/文本密度像素启发式、ensemble CLIP 做实拍↔资料二分与子类、dHash+时间分桶查重、有 GPS 才钉/无则待确认、IndexedDB 增量缓存(dHash 为键)。【必须诚实的 PWA 局限,不在文案承诺等同 Apple 精度】①HEIC+GPS:iPhone 默认 HEIC,exifr 在 PWA/Safari 上 GPS 解析失败率高——所以坐标做了降级而非硬闸,且建议落地前先做一次真机 HEIC+GPS 取数验证;②拿不到全相册(系统选择器逐次多选)、拿不到删权限(只能给建议清单引导用户去系统相册自己删,反而天然安全);③无系统级 isUtility/人脸/saliency/美学分,人物档与主体档是近似实现、默认偏保留;④iOS PWA 后台 JS 被冻结、requestIdleCallback 支持差→不承诺后台自动扫;⑤CLIP 镜像(hf-mirror)不稳/离线→isUtility 闸降级为不否决。【原生 app 以后升级,接口不变只换零件】截图→PHAsset.mediaSubtype .photoScreenshot(系统级0成本最准);资料/工具型→VNCalculateImageAestheticsScores 的 isUtility;美学分→同 API overallScore(-1..1);场景子类→VNClassifyImageRequest(约1303类层级 taxonomy);相似分组→VNGenerateImageFeaturePrint+computeDistance(从0.35起重标定);人脸/主体→Vision 人脸/saliency;文档/OCR→DetectDocumentSegmentation+VNRecognizeText;全相册+后台增量+NPU→PhotoKit change observer+BGProcessingTask;位置补位→Significant Locations(端到端加密)做时空借位。原则:所有 PWA 判别都是『建议+置信度+纠错记忆』,把每个启发式零件平滑替换为系统信号即升级。
