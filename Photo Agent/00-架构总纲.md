# 端侧整理照片 agent · 总架构（黄佳六层21模式 + Harness工程之道 + 机制 合成）

## 一句话
把现状 runScreen 那个『质量分单决留/钉』的一次性打分器，重构成感知→记忆→推理→行动→反思→协作的端侧具身闭环 Worker：Harness 四底座(白名单物理保证/舱壁降级/断点续跑/分级推理)做工程地基，三明治架构(确定性逻辑夹住端侧 CLIP)保可靠，verdict⟂pinnable 类型优先正交闸把质量分降级为只管废片过滤+簇内选优，全程 suggest-then-confirm 原图不出端。

## 设计哲学
三明治架构(确定性逻辑夹住概率模型，生成可概率/验证必确定) + 人机主导权(suggest-then-confirm，人 decision/AI choice/系统 execution) + 建议而非执行(绝不替用户删原图) + 类型优先正交闸(verdict⟂pinnable，质量分降级为只管废片过滤+簇内选优=连根拔除『质量分单决留/钉』翻车根因) + Harness 四底座(白名单物理保证/舱壁降级/断点续跑/分级推理省算省电) + 诚实对待 PWA 硬限制(无后台/无全相册/HEIC-GPS 读不出/无人脸 saliency)。核心立场转折：现状 runScreen 是把感知/记忆/推理/行动揉进一个大函数、还让 fastScore 单独决定留/钉(line 312-319) 的一次性打分器；要重构成六层具身闭环 Worker，per-photo 走串行流水线、阶段间只传结构化小结论(Handoff Contract)、原图/canvas/embedding 绝不出端也绝不回流主上下文。三套草案的『思想合成』全采纳；但它们对接现有代码有 4 处事实错误(钉地球链路是 0 不是『替换 addMoodSticker』；MAP_MARKERS 是 const 无运行时发布订阅；edgeSafe.vision 是网络调用违反原图不出端；FrostIntent/RadioAction 是封闭枚举 photos 无法路由)——已逐一核实并在本架构里改对：钉地球从零新建并复用 geoStickers 那套 localStorage 发布订阅、复用契约里已定义的 mark_place 而非另造 addPhotoPin、端侧 CLIP 走 photoScreen.ts 内 transformers.js 绝不走 edgeSafe.vision、FROST 接入诚实承认要动 types/router/validator 内核。

## 六层架构
### 感知
把一张 File 解码成带置信度的便宜信号，贵操作压到最后最少。per-photo 八闸认知漏斗：闸0 解码缩图(长边256，HEIC createImageBitmap+{imageOrientation:'from-image'} 归一化→<img>兜底)+算 dHash 主键 → 闸1 EXIF 金信号(最便宜最确定) → 闸2 像素废片闸(先过低光/艺术豁免) → 闸3 资料 isUtility 多信号投票 → 闸4 仅不确定(0.35~0.65)或将进钉候选才跑端侧 CLIP。每张独立舱室：HEIC 解码失败/EXIF 缺失/坏图只标 status:failed 跳过不中断整轮。多模态融合像素+EXIF+GPS 成统一表征。【修 OOM·评审实证 bug】现状 photoScreen.ts:277 给每一张 result 都挂 _canvas、全程驻留到 322 才 delete=大相册必 OOM；改为单张闸跑完即吐 PhotoFeatures 小结论、立即释放 canvas，buffer 绝不留主决策上下文。

**用到的模式**：认知漏斗/注意力聚焦(廉价评分器先分流90%废片资料，主动忽视而非全处理)；多模态融合/统一语义场(像素+EXIF+GPS 融合，从『看』到『在』)；三明治上层面包(像素/EXIF 确定性约束包住 CLIP 概率输入)；舱壁/优雅降级(单张坏图 status:failed 锁沙箱，不污染整轮)；分级推理/具身预算(System1 秒判→仅不确定项升级 System2 CLIP，按电量/充电下发精筛比例)

### 记忆
端侧持久化地基：已筛过别重筛、偏好沉淀、坐标治理。L1 工作记忆=当前批 File；L3=IndexedDB 照片索引(dHash 主键)。生命周期四步：编码(算 dHash+脱敏，只写派生)→巩固(去重写入)→检索(重扫先查 dHash 命中即秒级跳过，重活 CLIP/OCR 绝不重复跑)→遗忘(用户删的软删+坐标可清空)。RAG 式知识解耦：经纬度→地名当外挂(PLACE_COORDS 仅城市级标签『约』不作定位)。【诚实】PWA 系统选择器拿不到全相册全集，故『重扫只处理新增』退化为『用户重复喂入时去重』，无法做『稳定库秒级跳过整本』——那是原生 PhotoKit change observer 才有的能力，不在 PWA 文案承诺。

**用到的模式**：分层记忆/倒金字塔(L1 批×L3 IndexedDB 索引，只存派生元数据保持端侧轻盈)；对抗遗忘/高效遗忘(不存原图全量特征，只留指纹+派生)；情节记忆双键检索(目标相似+状态相似→既查重又举一反三归类)；RAG/知识解耦(坐标→地名外挂可更新，模型不背地理)；幂等主键(dHash 替换 line 272 跨批漂移/碰撞伪键)

### 推理
把 PhotoFeatures 推成三件套 photoType + verdict + pinnable。判定顺序固定『先类型后价值』。两条正交输出：verdict(留/待定/可清理 建议，压倒性偏『宁可放过』)由类型+价值+重复定；pinnable=实拍类型 ∧ 真坐标(或用户补) ∧ 价值达钉阈 ∧ 簇代表，资料型即便带坐标即便判留 pinnable 恒 false(类型一票否决优先于坐标)。质量分被彻底降级为只管废片过滤+簇内选优。价值按 value_by_type 算(风景=美学+地理稀有度，人物=有主体清晰，资料=可检索性非美)，稀有度分母=IndexedDB 累积全库，小库自动调低权重。

**用到的模式**：三明治下层面包/验证器(类型闸/坐标闸以 EXIF/像素确定性信号否决 CLIP 概率误判)；ReAct 微闭环(单张『像截图』→查 EXIF/GPS→观察→改判，设步数上限超限标待确认防死磕)；快/慢思考路由(System1 启发式秒判 / System2 CLIP 仅升级不确定项)；类型优先决策树(先分清是什么再谈值不值/钉不钉)；时空聚类(时间10分钟分桶→GPS DBSCAN eps 数百米，每簇=一次经历，增量只比相邻簇)

**模块**：

## 数据模型
主键=dHash(64bit 内容指纹，在感知层 decode 即算)，替换 photoScreen.ts:272 那个 'p'+i+'_'+(file.size%100000) 的非内容、跨批漂移/碰撞伪键。【已核实的两个 dHash 前置修正——评审遗漏点，必须先做否则主键照样漂移】① decode 的 createImageBitmap(line 56) 与 <img> 兜底(line 62) 都未传 imageOrientation/未做 EXIF orientation 归一化——同一张带旋转标志的 HEIC 走不同解码分支像素布局不同→dHash 漂移；故 dHash 必须算在「orientation 归一化后的灰度缩图」上(createImageBitmap 加 {imageOrientation:'from-image'})。② 64bit dHash 有碰撞概率，主键用 dHash 但写入/查重时以 (dHash,w:h 比例桶) 复合消歧，size/lastModified 仅辅助。存储=IndexedDB(库名 pe.photos)：store『photos』key=dHash，store『clusters』key=clusterId，store『progress』断点续跑游标，store『prefs』端侧个性化阈值(见 safety_privacy 为何不能进 profile.ts)。PhotoRecord(只存派生，schema 里根本没有原图/canvas/CLIP embedding 字段位置=物理保证)：{dHash(PK), photoType, verdict:'keep'|'review'|'clean', pinnable:boolean, isUtilityProb:number, hasGPS, lat, lng, source:'exif'|'user'|'borrowed', exif摘要{capDate,make,model,softwareIsScreenshot,suspectExif}, sharpness/exposure/colorful/contrast 四像素分, valueScore, clusterId, dupOf, userOverride, emotionalExempt:boolean, status:'ok'|'failed'|'pending', schemaVer, updatedAt}。ClusterRecord:{clusterId, repDHash(簇代表可被更好的替换), memberDHashes[], centroidLat/Lng(钉点坐标用簇心而非代表张坐标，避免代表刷新时点跳——补评审 gap), timeBucket, placeLabel}。现状 ScreenResult(photoScreen.ts:16-33) 是内存态(含 objectURL 给 UI)，落库时投影为 PhotoRecord；需扩 photoType/isUtilityProb/pinnable/hasGPS/lat/lng/clusterId/source/userOverride/emotionalExempt 字段。坐标是高敏轨迹：一键清空坐标/整库、默认不上云、分享脱敏。

## 决策(verdict ⟂ pinnable)
per-photo=有依赖的串行流水线(责任链/Unix pipe，黄佳《Harness》流水线型)，阶段间只传结构化小结论(Handoff Contract)，绝不把像素 buffer/embedding 往下传(端侧不 OOM 的生命线)：闸0 解码(orientation 归一化)+算 dHash 主键 → 闸1 EXIF 金信号(海里/出生前/截图工具名一致性校验标 suspectExif) → 闸2 像素废片闸(先过低光/艺术豁免：夜景/星空/虚焦/雪雾不判废) → 闸3 资料 isUtility 多信号投票(替换 line 270 脆弱单规则) → 闸4 仅不确定(0.35~0.65)或将进钉候选才跑端侧两段式 ensemble CLIP(先实拍↔资料二分纠偏 isUtility，实拍再判子类；CLIP 不可用时 isUtility 降级为不否决·宁放过，绝不让启发式单独误杀) → 闸5 查重(EXIF 时间分桶+海明≤2 严格/3~8 相似簇，资料放宽) → 闸6 时空 DBSCAN(把散点变经历) → 闸7 按类型算价值+簇内选代表+反查地名 → 闸8 出三件套+只写派生数据。决策核心=verdict⟂pinnable 双正交：verdict(留/待定/可清理)压倒性偏『宁可放过』——废片/相似簇非代表→可清理；实拍价值≥60 留/40~60 待定/<60 且确属像素废片才可清理；资料·拍摄文字默认待定·归档绝不默认可清理(可能是报销凭证/证件)；资料·屏内生成默认待定(纪念截图/逝者聊天被默认标可清理是比资料误留严重得多的反向伤害)；仅 dHash 严格重复且无情感信号才给弱可清理。pinnable=实拍 ∧ 真坐标 ∧ 价值达钉阈 ∧ 簇代表，资料型恒 false。质量分(line 125 fastScore)被彻底降级为只管废片过滤+簇内选优，绝不再单独决定留/钉(=连根拔除 line 312-319 用户翻车根因)。可并行的仅同张内独立子判定，端侧并行抢内存未必划算按设备能力定；查重/钉点有全局依赖必须串行。

## 隐私安全
四条硬约束全部落成架构层物理保证而非文案。① 原图/canvas/embedding 不出端：分析阶段工具集根本不放网络/上传工具(白名单即安全边界)，端侧 CLIP 走 photoScreen.ts 内 transformers.js 浏览器本地路径——【关键纠错·已核实：绝不用 edgeSafe.vision，它经 httpEdge 是网络调用(/api/edge)，用它跑 CLIP 会把图传出端，违反原图不出端；且 edgeSafe.vision 只返回字符串，拿不到 embedding，连『ensemble 余弦』都做不了】；canvas 单张跑完即释放(修 277/322 OOM)；IndexedDB schema 无原图/embedding 字段位置=无处可存。② 绝不替用户删原图：工具集无 deleteAsset/writeOriginal(能力不存在)，整会话 plan 只读模式；PWA 本就拿不到删权限，只能给建议清单引导用户去系统相册自己删(反而天然安全)。③ 进库只有派生小结论(dHash/类型/分数/verdict/pinnable/坐标/exif 摘要)。④ suggest-then-confirm：人 decision/AI choice/系统 execution，即便点清理也走移入待删相册+时间锁延迟而非物理删。⑤ 舱壁降级：单张坏图/HEIC 解码失败只标 status:failed 跳过不中断整轮；缺 GPS 不钉但仍完成分类。⑥ 坐标隐私(比原图不出端更被低估)：分享/截图地球时 source=exif 精确坐标抽稀到街区/城市级、家庭/公司高频点识别后默认不外泄、source=borrowed 借位坐标默认不外泄、一键清空坐标/整库、默认不上云。⑦ 情感豁免【诚实降级·补评审 gap：PWA 判不出人脸/人名/是否被收藏，故 PWA 阶段对资料一律默认 review 不 clean=靠『默认保守』兜底而非靠『豁免识别』；真正的含人脸/人名情感豁免推迟到原生 Vision，不在 PWA 文案空头承诺】。⑧ 端侧个性化阈值落 IndexedDB prefsStore，不进 profile.ts(其 line6-7 声明只供云脑、端侧 Selector 一律不接触)。

## 长期任务
增量+幂等+空闲跑+纠错闭环的前台流水线(黄佳《Harness》断点续跑)。① 稳定幂等键 dHash(orientation 归一化后算)：photoStore 记每张 dHash+结论+处理游标，重启读进度只处理新增、稳定项秒级跳过、重活 CLIP/OCR 绝不重复跑、手机切后台/锁屏/被杀也能断点续跑。② 分层省算省电：像素废片(秒)→多信号投票(便宜先滤资料)→仅不确定 top 跑 CLIP→仅资料跑 OCR(原生)；一次 embedding 复用类型分类+美学两任务；整批全是资料(报销季收据)快速短路直接进资料分区跳过钉地球与美学排序。③ 钉点增量：新过闸实拍点并入已有时空簇，簇只增不重算，簇代表可被更好的替换，钉点坐标用簇心(centroidLat/Lng)而非代表张坐标——代表刷新时点不跳(补评审 gap)。④ 纠错闭环：userOverride 写 IndexedDB+回写端侧 prefsStore 微调阈值。⑤【诚实纠正『后台持续生长』叙事——评审实证 gap：iOS PWA 后台 JS 被冻结、requestIdleCallback 支持差(降级 setTimeout 切片)、拿不到全相册/后台权限，故 PWA 阶段根本不是后台任务，而是『用户每次主动喂一批→前台 Page Visibility 空闲分批每批数十张不卡 UI→结果入库累积→多次选图地球叠加』；真正的后台增量扫描是原生 PhotoKit change observer+BGProcessingTask 才有，不在 PWA 承诺。】⑥ 稀有度分母随库规模自适应小库调低。
