# 一、本书（第4章）讲的 harness/agent 工程之道

## 核心命题：harness 工程的本质是「驾驭复杂性」

全章的母题只有一个——**上下文污染（Context Pollution）就是大模型时代的「内存泄漏」与「进程干扰」**。Claude 的 20 万 token 窗口虽大但不是无限，真正的杀手不是「容量不够」而是**信噪比失衡**：500 行日志里只有 3 行有用，200 行测试输出只需要「3 失败 47 通过」一句话。噪声不仅占空间，更**稀释注意力、诱发幻觉**（书里小冰的开场案例：Claude 遗忘关键错误码、误判无关失败项、引用不存在的函数名）。

作者给出的解法是把软件工程数十年沉淀的经典原则**迁移**到 agent 架构里。这是全书最重要的方法论姿态——agent 不是新物种，它本质就是一个复杂软件系统，必然遵循软件系统工程的一般规律。

## 1. 主循环/委派模型：CEO 心智，而非埋头职员

核心隐喻是「CEO 委派」：主 agent 不该自己吞下所有原始数据，而该**拆解任务、委派给专门的「员工」（子 agent）、只接收精炼汇报、基于汇报做决策**。这是一种「委派式思考」——成熟管理者界定每个任务的职责边界、能力要求、交付标准，而非事必躬亲。

## 2. 子 agent 的三大核心价值（对应三条软工原则）

- **隔离**（≈ 进程隔离 / 舱壁模式）：子 agent 有独立上下文窗口，读取的文件、命令输出、中间推理**绝不回流主对话**；噪声被封锁在「舱壁」内，主对话只收结论。
- **约束**（≈ 最小权限原则）：通过 `tools` 工具白名单构建**物理边界**，而非靠 prompt 里「请不要修改文件」这种易被忽视的口头建议。code-reviewer 只给 Read/Grep/Glob，即使 Claude 误判意图也无法写文件。
- **复用**（≈ 组件化）：子 agent 以 Markdown 定义，可团队共享、跨项目迁移，新人入职即用。

## 3. 工具的设计哲学：白名单即安全边界

- **最小权限**：精确授予「完成职责所必需」的工具，不多不少。
- **白名单 > prompt 约束**：物理层禁止 > 语言层劝阻。
- **细粒度约束**：不仅限定工具种类，还能限定参数，如 `Bash(npm test:*)` 只允许执行 `npm test:` 开头的命令。
- **plan 模式**：比工具白名单更强的**系统级只读承诺**，整个会话被标记为只读，从架构层杜绝写入风险。

## 4. 模型经济学：按任务难度选模型

`model` 字段是「被低估的成本杠杆」：模式化任务（格式检查、日志过滤）用 Haiku，深度推理（架构设计、安全审计）用 Sonnet/Opus。「不需要聘请首席架构师来执行单元测试。」

## 5. 五种子 agent 模式（能力谱系，由浅入深）

| 模式 | 软工对应 | 工具权限 | 适用 |
|---|---|---|---|
| **只读型** 安全观察者 | 最小权限 / 只读副本 | Read/Grep/Glob + plan | 审查、审计、架构分析 |
| **执行型** 高噪声处理器 | 外观模式（Facade） | +受限 Bash | 跑测试、日志分析（输入 >> 输出） |
| **并行型** 多专家工作流 | MapReduce | 各自隔离 | 多个**独立无依赖**维度同时剖析 |
| **流水线型** 串行处理链 | 责任链 / Unix pipe | **逐阶段动态调整** | 有明确阶段依赖（定位→修复→验证→报告） |
| **团队型** 自组织协作 | — | 长期存续 | 需多轮通信深度协调（成本最高） |

关键判据：
- 并行型的**严格前提**是子任务完全独立、无共享状态；一旦有依赖必须改用流水线型。
- 流水线型的基石是**交接契约（Handoff Contract）**——上游输出格式必须严格匹配下游输入（bug-locator 的「根本原因文件 / 修复方向」正是 bug-fixer 的必需输入）。
- 专业化收益：一个「全能」agent 在 3 个维度各 70 分，3 个「专才」在各自领域 90 分。
- **不要过度设计**：日常 90% 场景只用「只读型 + 执行型」就够；嵌套不超过 2 层，需要 3 层以上是任务分解有缺陷的信号。

## 6. 子 agent × Skills：身份与方法分离

- **子 agent 定义 Who/What/Where/Output**（战略：身份与目标）；**Skill 定义 How/With-What/By-Standard/Quality**（战术：方法与标准）。
- 方向 A（子 agent 预加载 Skill，`skills:` 字段）：角色驱动，子 agent 是主角，Skill 是它的「操作手册/扳手」。
- 方向 B（Skill 派生子 agent，`context:fork`）：知识/流程驱动，Skill 是主角，子 agent 只是一次性的「隔离执行容器」。
- 复用威力：同一个 `secure-coding` Skill 既给 bug-fixer 修复时用、也给 code-reviewer 审查时用。

## 7. Token 经济学：隔离往往**更省**而非更贵

因为 API 无状态，每轮都要重发完整上下文，10000 token 噪声进主对话后会被**重复传输 N 次**。把噪声关进子 agent，主对话只收 100 token 摘要。但作者很诚实：
- Prompt Caching 会削弱成本优势（缓存命中只收 10%），把节省率从 6.8% 拉回 1~2%。
- **所以成本并非子 agent 的核心价值**。真正不可替代的两点是：① **上下文窗口保护**（避免噪声挤占触发上下文压缩、导致关键信息被当「低优先级」丢弃）；② **响应质量提升**（纯净上下文 = 注意力不被稀释）。
- 决策准则：看**输入/输出体量比**。输入 >> 输出才值得隔离；输入 ≈ 输出（如改一个函数）直接在主对话做更高效。

## 8. 可观测、容错与优雅降级（实战注意事项）

- **CLAUDE.md 继承**：根目录 CLAUDE.md 对所有子 agent 生效；优先级 = 子 agent 定义 > 项目级 CLAUDE.md > 用户级。
- **「报文传输」而非「共享内存」**：子 agent 之间无法直接通信，必须由主 agent 提取 A 的结论、嵌入 B 的任务描述——所以输出**必须高度结构化**才能被准确提取转发。
- **中断恢复（断点续跑）**：子 agent 被中断则内存中间成果丢失。长任务应把中间产物**持久化到文件**（如 `.claude-work/log-analysis.md`），重启后读文件定位断点续跑，避免从头再来。
- **故障舱壁/优雅降级**：log-analyzer 遇到死循环、幻觉爆发或格式崩溃，故障被锁在沙箱内，不污染主对话——这就是系统韧性。
- **name/description 的工程含义**：`name` 是日志/调试/调用链的唯一键（可观测性锚点）；`description` 是路由依据，主 agent 靠它做**语义匹配→意图识别→自动委派**。

# 二、对「专业成熟的端侧整理照片 agent」的工程启发

你的照片 agent 的硬约束：手机端侧逐张分析、分类型/判价值/查重、把地方照片按经纬度钉回地球、长期本地跑、**隐私（原图不出端）**、**建议而非执行（绝不替用户删原图）**。下面把书里的原则逐条映射成可落地的设计。

## 1. 用「执行型 + 流水线」做单张照片的处理链（最直接的映射）

单张照片处理天然是**有依赖的串行链**，正是流水线型模式：

```
解码/取 EXIF → 像素特征 → 类型分类 → 价值判定 → 查重 → 地理钉回
   (locate)      (analyze)    (classify)   (value)    (dedup)   (geo-pin)
```

每一步是上一步的输入。**关键落地**：定义严格的「交接契约」——每阶段只输出**结构化小 JSON**（如 `{type, valueScore, isDuplicateOf, lat, lng, confidence}`），而不是把整张图的原始像素/特征张量往下传。这正是书里「报文传输而非共享内存」+「Handoff Contract」的应用，也是端侧内存吃紧时的生命线。

## 2. 信噪比优化 = 端侧 agent 的生存法则

书里「输入 >> 输出才值得隔离」对端侧加倍成立。一张 12MP 照片是几 MB 的原始输入，但你最终只需要一行结论。所以：
- **像素分析/小模型推理（高噪声、高内存）放进「执行型子任务」**，跑完只回吐一行摘要，**绝不把像素 buffer、embedding 全量留在主决策上下文里**——否则端侧内存直接 OOM。这对应书里「执行型 = 外观模式」，主循环只通过简洁接口拿结果。
- 主循环（统筹整理进度、给用户的建议列表）始终保持「纯净上下文」，不被逐张的中间数据污染。

## 3. 最小权限白名单 = 隐私与「建议而非执行」的**物理保证**

这是书里最该被你用满的一条。**不要靠 prompt 写「请不要删除原图」**——书里明说这种口头约束「易被忽视」。要靠**工具白名单构建物理边界**：

- 给分析阶段的 agent 只授予**只读**能力：读相册元数据、读缩略图/降采样像素。**根本不提供 `deleteAsset` / `writeOriginal` 这类工具**——能力不存在，幻觉也删不掉。
- 对应书里的 `permissionMode: plan`：让整个整理会话处于**系统级只读模式**，是比「我没给删除工具」更强的架构级承诺。
- 「原图不出端」同理用边界保证：分析 agent 的工具集里**根本没有「上传/网络请求」工具**，只有本地推理工具。即使模型被诱导也无网络出口。把它做成**端侧沙箱**——书里「即使子任务执行失败或产生恶意行为，影响也被限制在局部」。
- 「建议而非执行」落地为：所有「删除/合并重复」只产出一份**建议清单（proposal）**交给用户在 UI 里确认，agent 永远停在「生成建议」这一步。

## 4. 按任务难度分模型（端侧版的 model 字段）

端侧算力/电量是稀缺资源，书里「不需要首席架构师跑单元测试」对你就是省电省热：
- **粗筛（截图 vs 照片、模糊检测、纯色/文档判定、感知哈希查重）**用最轻的像素算法或最小模型，绝大多数照片在这一层就被分流。
- **只有需要语义理解的少数照片（判「值得保留的价值」、识别地标）**才上较大的端侧小模型。
这是把书里的「model 成本杠杆」翻译成端侧的「分级推理 pipeline」。

## 5. 断点续跑 = 长期本地任务的刚需

整理整本相册是**典型长任务**，且手机随时可能切后台、锁屏、被系统杀进程——完全命中书里「中断恢复」场景。落地：
- 维护一个**持久化进度文件/本地 DB**（类比书里 `.claude-work/log-analysis.md`），记录「已处理到第 N 张 / 各 asset 的 contentHash 与结论」。
- 重启后先读进度，**只处理新增和变更的照片**，跳过已分析的。
- 因为原图不出端、且 iOS PWA 拿不到全相册（你的 memory 里已记的硬限制），这套「内容哈希 + 增量」机制还能在每次只能拿到部分照片时做**幂等去重**，避免重复钉点、重复建议。

## 6. 舱壁/优雅降级 = 单张坏照片不能拖垮整轮整理

书里 log-analyzer 遇到「死循环、幻觉爆发、格式崩溃」被锁在沙箱内——端侧必然遇到坏文件、HEIC 解码失败、EXIF 缺失、损坏图。设计：
- 每张照片的处理是独立「舱室」，**单张失败只标记 `status: failed` 并跳过**，绝不中断整轮。
- 缺 GPS 的照片不钉地球但仍完成分类（部分结论优于全盘失败）——这就是书里的「优雅降级」「部分结论也是结论」。

## 7. 并行型用在哪、不用在哪

- **可并行**：同一张照片的「类型分类」「模糊/质量评估」「地标识别」彼此独立，可并行（MapReduce 的 Map）。但端侧要权衡——并行多个小模型可能反而抢内存/算力，**未必比串行划算**，按设备能力动态决定（书里也强调并行要「最大化利用资源」但有前提）。
- **不可并行**：查重依赖全局已处理集合、地理钉点依赖分类结果——这些有依赖的必须串行。

## 8. description/可观测：让整理过程可解释、可追溯

- 给每个处理阶段一个稳定 `name` 作为**日志锚点**，记录每张照片在哪一阶段、用了什么判据得出什么结论——用户问「为什么这张被建议删」时能给出可追溯的理由链（这正是书里 `name` 在调试追踪/调用链里的作用）。
- 结论必须**结构化**（书里反复强调），既为阶段间传递，也为 UI 直接渲染「建议卡片」。

## 9. 复用与可演进

把「类型分类规则」「价值评分标准」「查重阈值」「地理钉点逻辑」拆成**可复用的 Skill/配置模块**（书里 Skill 定义 How/Standard/Quality），与「照片整理 agent」这个身份（Who/What）分离。好处：调整价值评分标准只改一个文件、不动主流程；未来加「相似连拍择优」「人物聚类」等新能力时，是新增模块而非重写主循环。这对应书里「子 agent 定义身份、Skill 定义方法」的解耦。

## 一句话总结

这本书的 harness 工程之道，本质是把**进程隔离、最小权限、舱壁容错、责任链、MapReduce、外观模式**这些经典软工原则迁移到 agent 架构，核心目标是**用隔离与约束驾驭复杂性、守住信噪比与安全边界**。落到你的端侧照片 agent，最该照搬的三条是：**① 用流水线 + 严格交接契约处理单张照片并只在阶段间传结构化小结论（护住端侧内存与上下文）；② 用工具白名单 + plan 只读模式把「原图不出端」「绝不替用户删图」做成物理保证而非口头约束；③ 用持久化断点续跑 + 单张舱壁降级，让这个长期本地任务在随时被杀进程、随时遇到坏图的真实手机环境里稳定续命。**
